Three agents. One GPU. Shared prefix.

```
Shared root (system prompt + tool schemas): decoded once
  ├── Agent 0: fork → investigate question A
  ├── Agent 1: fork → investigate question B
  └── Agent 2: fork → investigate question C
```

Every agent inherits the root's KV cache at zero cost — an O(1) metadata tag, not a copy. Each agent then generates independently, calls tools, and reports findings. The pool manages the entire lifecycle.

## What is an Agent?

An agent is a **branch with intent**. A Branch is a forkable KV cache sequence — it stores every token the model has seen and generated. An Agent adds: a task to accomplish, a policy for how to accomplish it, and a record of what it has done.

The branch is the ground truth. The agent is the interpretation layer that gives meaning to what's in the branch and makes decisions about what to do next.

```typescript
import { Agent } from "@lloyal-labs/lloyal-agents";

// agent and agentPool return Agents.
// After completion, each agent exposes:
result.result          // what the agent discovered (or null)
result.rawOutput       // raw model output from the final turn
result.toolCallCount   // how many tools it called
result.tokenCount      // how many tokens it generated
result.nestedResults   // results from recursive sub-agents
result.ppl             // model-level perplexity at completion
```

### Status model

Every agent transitions through a lifecycle:

```
idle → active → awaiting_tool → active → ... → idle → disposed
```

- **idle** — created but not yet generating, OR finished but branch still alive (extraction window)
- **active** — generating tokens between stop boundaries
- **awaiting_tool** — tool call parsed, waiting for result in SETTLE
- **disposed** — branch pruned, agent no longer usable

The pool enforces valid transitions. An agent cannot go backward or skip states.

### Findings

Agents produce results through a single write path with provenance tracking:

- `report_tool` — agent voluntarily called `report()`
- `free_text` — agent emitted prose without a tool call
- `scratchpad` — extracted post-idle via grammar-constrained fork
- `tool_error` — tool threw, error captured as result

### Tool history

Each agent records metadata for every tool invocation — tool name, arguments, result token count, context available after settlement. The [Agent Policy](#agent-policy) reads this history for informed lifecycle decisions.

### Parent lineage

When a tool spawns sub-agents (recursive tools like `research` or `web_research`), each sub-agent records which agent called the tool that created it. `walkAncestors()` traverses the chain — self → caller → caller's caller — collecting data from each ancestor. Tool guards use this for cross-level deduplication.

## Single agent: `agent`

The simplest entry point. One agent, one task, automatic lifecycle:

```typescript
import { agent } from "@lloyal-labs/lloyal-agents";

const result = yield* agent({
  systemPrompt: "You are a research assistant.",
  task: "Find information about voice agent architectures.",
  tools: [searchTool, fetchTool, reportTool],
  terminalTool: "report",
});

console.log(result.result);
```

`agent` handles everything internally: builds the toolkit, creates a shared root, forks an agent branch, runs the tick loop, and prunes all branches when done. You declare what you want — the HDK orchestrates it.

### Structured output with `schema`

For single agents that should produce structured JSON instead of calling tools:

```typescript
const result = yield* agent({
  systemPrompt: "You analyze research queries. Output JSON only.",
  task: `Analyze: "${query}"`,
  schema: {
    type: "object",
    properties: {
      questions: { type: "array", items: { type: "object", properties: { text: { type: "string" }, intent: { type: "string" } }, required: ["text", "intent"] } }
    },
    required: ["questions"],
  },
});
const plan = JSON.parse(result.rawOutput);
```

The `schema` is compiled to a GBNF grammar and set as an eager constraint — every token is grammar-valid. No retry logic, no post-hoc validation.

### Warm path via `session`

When a Session has a trunk (previous turns committed), pass it to continue from the prior conversation state:

```typescript
const result = yield* agent({
  systemPrompt: EVAL_PROMPT,
  task: evalQuestion,
  schema: evalSchema,
  session,  // agent forks from session.trunk — inherits prior KV
});
```

## Parallel agents: `agentPool`

When a task decomposes into sub-questions, run them in parallel:

```typescript
import { agentPool, parallel } from "@lloyal-labs/lloyal-agents";

const pool = yield* agentPool({
  orchestrate: parallel(
    questions.map(q => ({ content: q, systemPrompt: RESEARCH_PROMPT })),
  ),
  tools: [searchTool, fetchTool, reportTool],
  terminalTool: "report",
  maxTurns: 20,
  policy: researchPolicy,
  pruneOnReport: true,
});

// pool.agents[0].result, pool.agents[1].result, ...
```

`agentPool` composes `withSharedRoot` + toolkit construction + the tick loop internally. The `orchestrate` callback declares the execution shape (`parallel`, `chain`, `fanout`, `dag`, or any custom `Orchestrator` generator); each `SpawnSpec` carries its own `content` + `systemPrompt`. Agents fork from a shared root, advance in the same GPU batch, and report independently.

### `agentPool` options, by significance

The `CreateAgentPoolOpts` type has many knobs but most of them have sensible defaults. Reach for them in roughly the order you'd ask the questions:

#### Required

- **`orchestrate`** — what shape (`parallel` / `chain` / `fanout` / `dag` / custom). Mandatory; no default makes sense.
- **`tools`** — what data access. Optional in the type but practically required (otherwise use `useAgent`).
- **`terminalTool`** — name string for the "I'm done" tool (e.g. `'report'`). Without it, agents exit via free-text-stop or recovery extraction — both heavier and less clean. Practically required for any harness with non-trivial tool use.

#### "What about follow-up questions?" — `session`

`Session` (from `@lloyal-labs/sdk`) is **the** multi-turn knob. It owns the **trunk** — a long-lived KV branch that persists across queries within the same model context. After a pool exits, the harness calls `session.commitTurn(query, answer)` and the trunk is extended with that Q&A turn. The next pool forks from the (now extended) trunk and the agent reads prior Q&A via KV attention.

```typescript
import { Session } from "@lloyal-labs/sdk";

const session = new Session({ ctx, store });

// First query
yield* agentPool({ orchestrate, tools, session });
yield* call(() => session.commitTurn(query1, answer1));

// Follow-up — agents fork from session.trunk, see prior Q&A
yield* agentPool({ orchestrate: chain([followup]), tools, session });
yield* call(() => session.commitTurn(followup, answer2));
```

A `Session` IS the persistent state. Not a message array that gets re-tokenized — a literal KV branch on the same model context. Cold start = no trunk, pool creates a fresh root at position 0. Warm = trunk exists, pool forks from it. The trunk is the only thing that survives the harness's lifetime; everything else (research branches, sub-pools, delegations) is pruned on scope exit.

If `session.trunk` is set and you don't pass `parent`, `agentPool` forks from the trunk automatically. If you pass both, `parent` wins (use this for sub-pools that need to fork from a specific calling agent's branch).

#### "What about sub-agents?" — `parent`

For warm-path forking: `agentPool({ parent: context.branch })`. The new pool's root forks from the calling agent's branch instead of cold-starting. Sub-agents inherit the parent's full attention state — every tool call, every result, every token. `DelegateTool` from `@lloyal-labs/rig` uses this internally; you'd reach for it directly when threading roots between harness stages.

Precedence: `parent ?? session?.trunk ?? cold-start`.

#### "How do I tune budgets and recovery?" — `policy`

The default `DefaultAgentPolicy` is fine to start. You'll customize it when:
- The default time budget (`time: { softLimit: 120_000, hardLimit: 180_000 }`) is too tight for your tool calls.
- You want a custom recovery prompt for `recoverInline` extraction.
- You want extra tool guards (the default has dedup guards for `web_search` / `fetch_page` and a research-before-delegation gate).

```typescript
const policy = new DefaultAgentPolicy({
  budget: {
    context: { softLimit: 2048, hardLimit: 1024 },
    time: { softLimit: 240_000, hardLimit: 360_000 },
  },
  recovery: { prompt: REPORT_PROMPT },
});

yield* agentPool({ orchestrate, tools, policy });
```

#### "How long do agents run?" — `maxTurns`

Default 100 — almost always too generous. Research pools cap at 10. This is a per-agent turn cap, not pool-wide. A "turn" is one tool-call → result-settle round.

#### "I have many concurrent agents — KV is filling up" — `pruneOnReport`

Set `true` for parallel pools where each agent's branch grows large and you want sibling agents to recover headroom mid-flight. The agent's branch is pruned immediately after the terminal tool fires. Spine extensions (which live on a separate root) are unaffected.

#### "I have many agents sharing the same tool palette" — `systemPrompt` (catalog mode)

When set, `agentPool` runs in catalog mode: the chat header (system + tool schemas) is prefilled onto an inner queryRoot once at setup; every agent inherits via metadata-only KV prefix-share instead of re-emitting the schemas in its own suffix. See [Skill Catalog](/reference/skill-catalog) for the full convention and tradeoffs (notably: in catalog mode, `extendRoot` writes to the inner root, not the warmParent).

For chains where you want spine extensions to persist past pool exit (so post-pool synth can attend to them), put the catalog on the **harness's** outer `withSharedRoot`, not on `agentPool` directly.

#### "I'm running a thinking-mode model" — `enableThinking`

Default `false`. Set `true` for Qwen3 thinking-mode and similar; the chat template emits `<think>` block delimiters and the parser handles them as reasoning content (separate from the tool-call stream).

#### RIG-specific knobs

- **`scorer`** — entailment scorer (from a `Source.createScorer(query)`). Used by tools (search reranking, DelegateTool's echo gate) for semantic coherence checks. Propagates through `ToolContext` to all recursion depths.
- **`echoThreshold`** (default 0.8) — cosine threshold for the delegation echo gate. Sub-questions whose similarity to the parent task exceeds this get rejected before sub-agents spawn — prevents degenerate recursion where each level re-asks the same question in different words.
- **`checkAncestorEcho`** (default false) — whether echo detection walks the full ancestor chain or only the immediate parent.

#### Diagnostic

- **`trace`** — when true, emits per-token entropy and surprisal on `agent:produce` events. The TraceWriter itself is set globally via `initAgents`.



### Per-spec system prompts

Each spec carries its own system prompt — they're required, not optional:

```typescript
const pool = yield* agentPool({
  orchestrate: parallel(
    questions.map((q, i) => ({
      content: q,
      systemPrompt: renderPromptForAgent(q, i, questions),
    })),
  ),
  tools: [searchTool, reportTool],
  terminalTool: "report",
});
```

Per-spec prompts are formatted independently by each agent's `setupAgent` call — the shared root provides the KV prefix, individual agents diverge after fork.

If you want a single role + tool palette amortized at the root once across N agents, opt into the **catalog convention**: pass `systemPrompt` on `agentPool` (or, more typically, on a wrapping `withSharedRoot`). See [Skill Catalog](/reference/skill-catalog).

### Recursive agents (delegation)

For agents that spawn sub-pools, use `DelegateTool` from `@lloyal-labs/rig`. It's a tool: pass an instance into the parent agent's toolkit, and when the model calls it, the tool itself runs `agentPool` internally — sub-agents inherit the calling agent's full KV state via Continuous Context.

```typescript
import { DelegateTool } from "@lloyal-labs/rig";

const delegate = new DelegateTool({
  name: "web_research",
  description: "Spawn parallel sub-agents to investigate sub-questions.",
  argsSchema: {
    type: "object",
    properties: { questions: { type: "array", items: { type: "string" } } },
    required: ["questions"],
  },
  extractTasks: (args) => args.questions as string[],
  systemPrompt: RESEARCH_PROMPT,
  poolOpts: {
    tools: [searchTool, fetchTool, reportTool],
    terminalTool: "report",
    scorer,
  },
});

const pool = yield* agentPool({
  orchestrate: parallel(
    questions.map(q => ({ content: q, systemPrompt: RESEARCH_PROMPT })),
  ),
  tools: [searchTool, fetchTool, reportTool, delegate],
  terminalTool: "report",
  scorer,   // propagated to all depths via ToolContext
});
```

Recursion depth is bounded by KV pressure, not by a hardcoded limit.

### Recovery

When agents are killed by KV pressure before calling `report()`, the policy extracts partial findings. Configure `recovery` on the policy:

```typescript
const researchPolicy = new DefaultAgentPolicy({
  recovery: {
    prompt: { system: "Call report with your findings.", user: "Report." },
    minTokens: 100,     // skip extraction below this (default 100)
    minToolCalls: 2,     // skip extraction below this (default 2)
  },
});
```

Recovery prefills a nudge prompt into the agent's existing branch and forces an eager `{ result: string }` grammar. The agent's full conversation context is still in KV — the grammar forces structured extraction from it.

### `pruneOnReport`

When `true`, agents are pruned immediately after reporting. This frees KV mid-pool, giving remaining agents more headroom. Without it, all branches stay alive until the pool completes.

## `useAgent` and `useAgentPool`

`agent` and `agentPool` are the scoped wrappers — branches are pruned when they return. When you need branches to stay alive after the pool completes (for forking, verification, or follow-up):

```typescript
import { useAgent, useAgentPool } from "@lloyal-labs/lloyal-agents";

// useAgent — resource, branch stays alive
const agent = yield* useAgent({
  systemPrompt: "...",
  task: "...",
  tools: [...],
});
// agent.branch is alive — fork from it for verification
const verifyBranch = agent.branch.forkSync();
// Branch pruned when the enclosing scope exits
```

`useAgentPool` returns a `Subscription<AgentEvent, AgentPoolResult>` — events stream during execution, result is the close value:

```typescript
const sub = yield* useAgentPool({
  root,
  orchestrate: parallel(specs),
  toolsJson: toolkit.toolsJson,
  tools: toolkit.toolMap,
  // ...
});
let next = yield* sub.next();
while (!next.done) {
  handleEvent(next.value);  // AgentEvent
  next = yield* sub.next();
}
const pool = next.value;     // AgentPoolResult
```

Most code never uses these directly — `agent` and `agentPool` handle the Subscription drain and scope cleanup internally.

## Agent Policy

The pool consults an **AgentPolicy** at every decision boundary — when an agent hits a stop token, when deciding whether to kill under pressure, and when checking tool guards before dispatch.

```typescript
interface AgentPolicy {
  onProduced(agent, parsed, pressure, config): ProduceAction;
  shouldExit?(agent, pressure): boolean;    // trailing stop kill gate
  shouldExplore?(agent, pressure): boolean; // optional — defaults to true
  onRecovery?(agent): RecoveryAction;       // scratchpad extraction
  resetTick?(): void;                       // per-tick flag reset
}
```

`DefaultAgentPolicy` handles all the default behavior:

- **Terminal tool interception** — when the agent calls `report()`, extract result and mark idle
- **Budget enforcement** — deny non-terminal tool calls when over budget, nudge the agent to report
- **Confabulation prevention** — reject premature reports from agents with fewer than 2 tool calls
- **Tool guard checking** — run declarative guards before dispatching any tool call
- **Explore/exploit switching** — decide per-DISPATCH whether content-boundary tools should explore (agent-local scoring) or exploit (dual scoring against original query)

Customize thresholds and guards via the constructor:

```typescript
import { DefaultAgentPolicy } from "@lloyal-labs/lloyal-agents";

const policy = new DefaultAgentPolicy({
  minToolCallsBeforeReport: 3,
  extraGuards: [myCustomGuard],
  exploreThreshold: 40,
  budget: { context: { softLimit: 2048 }, time: { softLimit: 600_000 } },
  recovery: { prompt: REPORT_PROMPT },
});

const pool = yield* agentPool({
  orchestrate: parallel(specs),  // each spec carries its own systemPrompt
  tools,
  policy,
});
```

The policy sees the agent's full state: status, tool history, result, nudge count, pressure snapshot. It returns an action — `tool_call`, `report`, `nudge`, or `idle` — and the pool executes it.

### Tool Guards

Guards are declarative rejection rules checked before tool dispatch:

```typescript
interface ToolGuard {
  tools: string[];
  reject: (args, lineageHistory, agent) => boolean;
  message: string;
}
```

Default guards:

1. **Fetch dedup** — rejects `fetch_page` if the same URL was already fetched
2. **Search dedup** — rejects `web_search` if the same query (case-insensitive) was already searched
3. **Research-before-delegation** — rejects `web_research` / `research` unless the agent has done at least one search AND one fetch itself

### Entailment scoring

When agents recurse, their local hypotheses can drift from the original query. The reranker (a 0.6B cross-encoder with its own context, zero inference KV cost) acts as a real-time entailment oracle:

| Method | Scores against | Purpose |
|--------|---------------|---------|
| `scoreEntailmentBatch(texts)` | Original query | Filter drifted delegation questions |
| `scoreRelevanceBatch(texts, localQuery)` | min(tool query, original query) | Tighten focus in exploit mode |
| `scoreSimilarityBatch(reference, texts)` | Arbitrary reference | Echo detection — reject paraphrased delegations |

The scorer propagates through the entire recursion tree — every depth scores against the same original query.

## Context pressure

The pool monitors KV cache usage every tick and enforces two thresholds:

- **Soft limit** (default 1024 tokens) — headroom reserved for downstream work. When exhausted, non-terminal tool calls are rejected.
- **Hard limit** (default 128 tokens) — crash-prevention floor. Agents are killed before `produceSync()`.

Pressure management is automatic. You don't check headroom in your tools. The pool drops agents gracefully when the context fills — agents that reported results keep them, agents that didn't get recovered via `policy.onRecovery()`.

## Prefix sharing

Every agent forks from a shared root branch — that's the cheap part (`Branch.forkSync` is metadata-only). What gets prefilled onto the root, and inherited via KV prefix-share, depends on whether you opt into the catalog convention:

- **Default (no `systemPrompt` on `withSharedRoot` / `agentPool`):** the shared root holds only a turn separator. Each agent's `setupAgent` formats its own `[system + tools + user]` suffix and tokenizes that onto its branch. Tools are re-emitted per spawn — fine for small pools, wasteful for many siblings sharing the same tool palette.
- **Catalog mode (`systemPrompt` + `toolsJson` on `withSharedRoot`):** the root is prefilled with the chat header (system + embedded tool schemas) once. Every agent inherits those tokens via metadata-only KV share. Agent suffixes carry only the per-spec `[system, user]` (a thin role assignment + task), no tools. See [Skill Catalog](/reference/skill-catalog).

```
Default:        3 agents x ~900 tokens of prompt+tools = 2,700 unique cells
Catalog mode:   ~900 shared root + 3 x ~30 per-agent suffix = ~990 unique cells
Savings:        ~64% fewer decode operations
```

When a `DelegateTool` spawns sub-agents, the inner `agentPool` forks from the calling agent's branch (warm path) instead of creating a new root. Sub-agents inherit every tool call, every tool result, every token the parent generated — **2 tokens instead of ~760** for the prefill.

<Tip>
For the full picture — KV layout, position-aware decode, nested sharing, measured numbers — see [Prefix Sharing](/reference/prefix-sharing).
</Tip>

## Nested results

When agents delegate to sub-agents via a `DelegateTool`, inner results bubble up through `nestedResults`:

```typescript
const pool = yield* agentPool({
  orchestrate: parallel(specs),
  tools: [...baseTools, new DelegateTool({ ... })],
});

// Primary findings — what each agent reported
pool.agents[0].result

// Nested results — what inner sub-agents found
pool.agents[0].nestedResults  // ["Sub-agent found X", "Sub-agent found Y"]
```

Any tool that returns `{ results: string[] }` or `{ nestedResults: string[] }` automatically participates — no tool-name hardcoding.

## Events

The pool emits events through the `Events` channel for every lifecycle transition:

- `agent:spawn` — agent created and prefilled
- `agent:produce` — token generated (with optional entropy/surprisal)
- `agent:tool_call` — tool invoked
- `agent:tool_result` — tool result received (with `contextAvailablePercent`)
- `agent:tool_progress` — progressive tool feedback (e.g. reranker scoring)
- `agent:report` — agent submitted result via terminal tool
- `agent:done` — agent completed
- `agent:tick` — batch decode committed (with KV pressure snapshot)

Subscribe in a spawned consumer to build TUI, logging, or monitoring:

```typescript
yield* spawn(function*() {
  for (const ev of yield* each(events)) {
    if (ev.type === "agent:report") {
      console.log(`Agent ${ev.agentId}: ${ev.result}`);
    }
    yield* each.next();
  }
});
```
