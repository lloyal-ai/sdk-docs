Agent pools share a single KV cache. As agents generate tokens and prefill tool results, the cache fills. `ContextPressure` tracks how much room remains and enforces shutdown thresholds at four points in the tick loop to prevent out-of-memory crashes. When all agents are killed by pressure, a one-shot recovery protocol extracts their findings before the pool closes.

## The budget model

`ContextPressure` reads from `SessionContext._storeKvPressure()`, which returns `{ nCtx, cellsUsed, remaining }` where `remaining = nCtx - cellsUsed`. Two caller-configurable thresholds partition `remaining` into three zones:

```
               nCtx
 ┌──────────┬───────────────────┬──────────┬────────┐
 │cellsUsed │    headroom > 0   │ softLimit│hardLim │
 │ (in use) │   (new work OK)   │(reserved)│(recov) │
 └──────────┴───────────────────┴──────────┴────────┘
              <- remaining ->
                                |          |
 headroom = remaining - softLimit          |
 critical = remaining < hardLimit ---------┘
```

- **headroom > 0** -- room for new work. Tool results can be prefilled, agents can continue generating.
- **headroom ≤ 0** -- over budget. SETTLE rejects tool results. PRODUCE denies non-terminal tool calls. Terminal tools (e.g. `report`) still pass through.
- **critical** -- `remaining` has dropped below `hardLimit`. Agents are killed before `produceSync()` to prevent `llama_decode` crashes. The `hardLimit` reserve is used by recovery extraction.

## `ContextPressure` API

The class is constructed from a `SessionContext` and thresholds from the policy. It captures an immutable snapshot of KV state at construction time -- frozen at phase boundaries so that all decisions within a phase are evaluated against the same baseline. Without this, items processed earlier in a loop would see different pressure than items processed later, making reject/nudge/kill decisions order-dependent and nondeterministic.

The pool reads thresholds from `policy.pressureThresholds` (which comes from `DefaultAgentPolicyOpts.budget.context`). You configure them on the policy, not the pool:

```typescript
const policy = new DefaultAgentPolicy({
  budget: { context: { softLimit: 1024, hardLimit: 2048 } },
});

pressure.nCtx              // total KV cache capacity (0 when no limit)
pressure.cellsUsed         // KV cells currently in use (monotonic within a pool run)
pressure.remaining         // nCtx - cellsUsed (Infinity when nCtx <= 0)
pressure.headroom          // remaining - softLimit
pressure.critical          // remaining < hardLimit
pressure.canFit(n)         // n <= headroom
pressure.percentAvailable  // 0–100 integer: remaining / nCtx * 100
```

`percentAvailable` is the single source of truth for the KV availability percentage. It is used by `policy.shouldExplore()` to decide explore vs exploit mode, recorded on `ToolHistoryEntry.contextAfterPercent` after each tool result settles, and injected into tool results as `_contextAvailablePercent` so agents can make pressure-aware decisions.

Defaults: `softLimit = 1024`, `hardLimit = 2048`.

## Four enforcement points

`ContextPressure` is instantiated at multiple points in the tick loop. Each snapshot is fresh -- constructed from current `cellsUsed` at that moment.

### 1. Prefill gate (pool setup)

Before the tick loop starts, the pool computes the total suffix cost for all agents -- the tokenized chat template (system prompt + tool schemas + user message + generation prompt) per agent. If the total exceeds headroom, agents are dropped from the end of the task list until the remainder fits:

```typescript
const initPressure = new ContextPressure(ctx, pressureOpts);
const totalSuffix = prefillSetup.reduce((s, [, t]) => s + t.length, 0);

if (!initPressure.canFit(totalSuffix)) {
  while (prefillSetup.length > 0) {
    const needed = prefillSetup.reduce((s, [, t]) => s + t.length, 0);
    if (initPressure.canFit(needed)) break;
    prefillSetup.pop();
    const dropped = agents.pop();
    dropped.state = 'done';
  }
}
```

This is the primary mechanism that limits agent count in nested pools. When an outer agent's tool spawns an inner pool, the outer agents' live branches consume KV that the inner pool's prefill gate measures against. An inner pool requesting 3 agents may only get 1 or 2 if the outer pool has consumed most of the headroom.

Agents dropped at the prefill gate never emit `agent:spawn` events -- they silently disappear before generating a single token.

### 2. PRODUCE critical (hard kill)

At the start of each tick, a fresh `ContextPressure` snapshot is taken. Any agent whose `pressure.critical` is true is killed immediately -- before `produceSync()` is called:

```typescript
const pressure = new ContextPressure(ctx, pressureOpts);

for (const a of agents) {
  if (a.state !== 'generating') continue;

  if (pressure.critical) {
    a.state = 'done';
    continue;
  }

  const { token, text, isStop } = a.branch.produceSync();
  // ...
}
```

The `hardLimit` defaults to 2048 tokens -- enough for recovery extraction (see below). If you reach this threshold, agents are killed and the reserved tokens are used for finding extraction.

### 3. PRODUCE soft-cut (non-terminal denial)

When an agent hits a stop token and the parsed output contains a tool call, the pool checks whether the call should be allowed:

```typescript
const overBudget = (a.turns >= maxTurns || pressure.headroom < 0)
  && (!terminalTool || tc.name !== terminalTool);
```

If `headroom < 0` (over the soft limit) and the tool call is not the terminal tool, the policy nudges or kills the agent. Terminal tools always pass -- an agent that has done research and wants to report should never be blocked by pressure, since the report call itself consumes minimal KV.

This is the mechanism that prevents agents from starting new work when the cache is nearly full. An agent that has accumulated 6 tool results worth of KV and wants to call `research` (which would spawn sub-agents consuming even more KV) gets denied. The same agent calling `report` goes through.

### 4. SETTLE deferral (tool result gating)

After DISPATCH executes tools, the next tick's SETTLE phase drains those results. Each result is pressure-gated — but SETTLE never kills agents. Items that don't fit are deferred until headroom recovers:

```typescript
const settlePressure = new ContextPressure(ctx, pressureOpts);
let headroom = settlePressure.headroom;

for (const item of items) {
  const a = agentById.get(item.agentId);
  if (!a || a.status === 'idle') continue;

  if (item.prefillTokens.length > headroom) {
    deferred.push(item);  // retry next tick
    continue;
  }

  prefillPairs.push([a.branch, item.prefillTokens]);
  headroom -= item.prefillTokens.length;
}

return deferred;  // caller carries forward to next tick's SETTLE
```

The `headroom` variable is tracked locally because `cellsUsed` doesn't update until `store.prefill()` at the end of the phase. The local variable accounts for items queued but not yet prefilled, ensuring all items are evaluated against the same baseline.

Tool results are the largest KV consumers. A `web_search` result can be 2,000+ tokens. A `fetch_page` result can be 6,000+. SETTLE deferral is the primary mechanism for backpressure during active research.

Deferred items stay in `pendingSettled` until headroom recovers. Headroom recovers when the PRODUCE trailing stop kills an agent, recovers its findings inline, and prunes its branch — freeing KV cells. Next tick, SETTLE retries the deferred items against the recovered headroom.

If all agents are `awaiting_tool` with deferred items and no active agents remain, a stall-breaker sacrifices one agent to free KV. Without this, agents with oversized results would wait indefinitely — PRODUCE skips `awaiting_tool` agents, so headroom never recovers.

### `_contextAvailablePercent` in tool results

After each tool executes, the pool creates a fresh `ContextPressure` snapshot and injects `percentAvailable` into the tool result object as `_contextAvailablePercent`:

```json
{"matches": [...], "totalMatches": 4, "_contextAvailablePercent": 32}
```

The value is derived from `ContextPressure.percentAvailable` -- the single formula for KV availability percentage across the system. The same value is emitted on the `agent:tool_result` event for TUI display and recorded on `ToolHistoryEntry.contextAfterPercent` for policy decisions.

## Recovery extraction (trailing stop)

When an agent is killed by pressure, the pool extracts its accumulated findings **inline** before continuing the tick loop. This is the trailing stop -- one agent sacrificed per critical tick, its findings extracted and KV freed, so remaining agents can continue researching.

### How it works

When `shouldExit` kills an agent during PRODUCE:

1. **Policy check**: Call `policy.onRecovery(agent)`. If it returns `{ type: 'skip' }`, prune the branch and continue.
2. **Pressure check**: If `remaining < promptTokens` (extraction prompt doesn't physically fit), skip recovery and prune.
3. **Grammar**: Create an eager JSON grammar from `jsonSchemaToGrammar({ result: string })`. Constrains from token 0 -- the model MUST produce `{"result": "..."}`.
4. **Prefill**: Prefill the extraction prompt into the agent's own branch. No fork -- the agent's full KV context (system prompt, tool results, reasoning) is physically present.
5. **Set grammar**: `branch.setGrammar(reportGrammar)`. Eager mode -- active from the first generated token.
6. **Generate**: Single-agent produce/commit loop -- `produceSync()` + `store.commit()` per token until stop token.
7. **Parse + report**: Parse the JSON output, call `reportResult(parsed.result, 'scratchpad')`.
8. **Prune**: Free the agent's branch. The freed KV cells give remaining agents headroom.

Recovery runs inside `scoped()` with error containment -- if `llama_decode` fails during the produce/commit loop (KV exhaustion), the scope tears down cleanly and the pool continues. This mirrors the `scoped()` + try/catch pattern used by tool dispatch.

<Note>
Recovery uses the agent's **own branch**, not a fork. The agent's full KV context is already there -- all tool results, all generated reasoning. The extraction prompt is appended directly. This eliminates the overhead of forking and the RESTRICT prune conflicts that forks create during scope teardown.
</Note>

### Why eager grammar, not lazy

During normal generation, agents use lazy grammar with tool-call triggers -- the model generates freely until `<tool_call>` fires, then the grammar constrains the tool call structure. This gives the model freedom to call any tool.

Recovery uses eager grammar (`setGrammar`, not `setGrammarLazy`). The JSON schema grammar constrains from token 0: the model must produce `{"result":` followed by findings text. No tool calls possible, no model choice. This is the system extracting findings, not asking the model to report.

The eager grammar also overrides thinking tokens. Even when the extraction prompt ends with `<think>\n\n</think>\n\n` (from `enableThinking: false`), the grammar constraint is structural -- applied before the sampler chain -- and masks everything except valid JSON tokens.

### Recovery and hardLimit

The `hardLimit` reserve exists specifically for recovery. Agents are killed when `remaining < hardLimit`. The trailing stop only needs to recover one agent at a time, so the reserve only needs to fit one extraction prompt (~65 tokens) plus generation headroom. Default `hardLimit` is 1024 tokens.

The recovery pressure check uses `remaining < promptTokens` (raw capacity), not `canFit` or `critical`. This is deliberate: the `hardLimit` reserve was set aside precisely for this moment. Checking `critical` would always fail because agents were just killed at the critical threshold.

### Trailing stop behavior

The policy's `shouldExit` kills at most one agent per tick when pressure is critical. The `_killedThisTick` flag prevents mass kills. After recovery extracts and prunes, the freed KV gives remaining agents headroom to continue. If pressure stays critical, another agent is killed next tick.

This is the sole kill path under pressure. SETTLE defers oversized results without killing. PRODUCE soft-cut nudges without killing (unless critical). Only `shouldExit` in PRODUCE kills, and only one per tick.

### One-shot guard (post-loop)

If all agents are idle without results after the tick loop ends (e.g., killed by `maxTurns` or `free_text_stop`), a separate one-shot recovery pass extracts from any remaining unrecovered agents. This fires at most once per pool run via the `recoveryAttempted` flag.

### Scope

Recovery is per-pool. In multi-source research, each source's pool runs its own recovery independently. The bridge phase between sources reads recovered findings from `pool.agents[].result`, so the next source's agents benefit from prior recovered findings.

## `cellsUsed` tracks live pressure

<Note>
`cellsUsed` increments on every `decode_each` (COMMIT, +1 per branch) and `decode_scatter` (SETTLE prefill, +tokens per branch). It **decrements** when a branch is pruned -- each branch subtracts its unique cells (`position - fork_head`) on release. It resets on bulk operations: `retainOnly` sets it to the winner's position, `drain` resets to zero.

`cellsUsed` can exceed `nCtx` -- there is no pre-decode guard in the native layer. Protection is entirely in the JS layer via `ContextPressure.critical`, which kills agents before `produceSync()`. If an agent slips through (e.g., during recovery), `llama_decode` will fail with "failed to find a memory slot for batch" and throw.

On `llama_decode` failure, `cellsUsed` is NOT rolled back (it remains at the post-increment value), but the branch's sampler, grammar, and metrics state IS rolled back via the native snapshot/restore mechanism.
</Note>

`remaining` and `percentAvailable` reflect actual KV usage and **recover** as branches are freed. When an agent reports and its branch is pruned (`pruneOnReport: true`), the freed cells are visible to the next `ContextPressure` snapshot. Remaining agents see increased headroom.

This has a direct impact on explore/exploit mode: `percentAvailable` can rise above the `exploreThreshold` after branches are pruned, switching surviving agents back to explore mode. The flip is intentional -- it produces higher quality trajectories by allowing discovery when headroom recovers.

## Sizing thresholds

The `hardLimit` must reserve enough tokens for recovery extraction:

| Component | Tokens |
|-----------|--------|
| Extraction prompt per agent (turn separator + system/user) | ~65 |
| 3 agents extraction prompts | ~195 |
| Grammar-constrained report generation per agent | 200-600 |
| Safety margin | ~200 |
| **Total** | **~2000** |

The `softLimit` must reserve enough for the nudge/report cycle: a nudge error injection (~30 tokens), one generation turn (~200 tokens), and a report tool call (~100 tokens).

For nested pools, the outer pool's thresholds must also account for the inner pool's entire lifecycle.

<Tip>
For a single-level recursive tool with up to 3 inner agents and `maxTurns: 6`, a `softLimit` of 2048-3072 provides adequate headroom for one level of nesting. For deeper recursion, each level needs its own budget reserved by the level above.
</Tip>

```typescript
const policy = new DefaultAgentPolicy({
  budget: { context: { softLimit: 3072, hardLimit: 2048 } },
});

yield* useAgentPool({
  tasks,
  tools: toolMap,
  maxTurns: 8,
  policy,
});
```

Monitor with `trace: true` -- the `[SETTLE]` and `[PRODUCE]` pressure logs print `remaining`, `headroom`, `cellsUsed`, and `nCtx` at each tick. Recovery logs `[RECOVERY]` with prefill count and remaining capacity:

```
[PRODUCE] SOFT_LIMIT remaining=2200 headroom=-104 cellsUsed=14184 nCtx=16384
[SETTLE] REJECT read_file:1059 > headroom=-104
[RECOVERY] prefilling 3 agents, 195 tokens, remaining=2188
[RECOVERY] prefill done, setting grammar
[RECOVERY] starting produce/commit loop
[RECOVERY] no entries, done=3/3
```

## Pressure-driven agent lifecycle

<Steps>
  <Step title="Pool setup">
    Prefill gate drops agents that don't fit above softLimit.
  </Step>
  <Step title="Each tick: PRODUCE critical check">
    Kill agents below hardLimit (critical). Reserved tokens protect recovery.
  </Step>
  <Step title="Each tick: PRODUCE soft-cut">
    Deny non-terminal tool calls when headroom &lt; 0. Terminal tools pass.
  </Step>
  <Step title="Each tick: SETTLE deferral">
    Defer tool results that exceed headroom. Retry next tick after KV is freed.
  </Step>
  <Step title="Trailing stop + recovery">
    One agent killed per critical tick → findings extracted inline → branch pruned → freed KV lets remaining agents continue.
  </Step>
</Steps>

Agents that survive the prefill gate will generate tokens until one of three things happens: they complete naturally (stop token + no tool call, or terminal tool call), they hit `maxTurns`, or the trailing stop in PRODUCE kills them. SETTLE never kills agents -- it defers oversized results until headroom recovers. The sole kill path under pressure is `shouldExit` in PRODUCE, which kills at most one agent per tick.

The `pool:agentDrop` trace event records the specific enforcement point that triggered the drop: `pressure_init`, `pressure_critical`, `pressure_softcut`, `pressure_settle_reject` (stall-breaker only), or `maxTurns`. Recovery prefills appear as `branch:prefill` with `role: 'recovery'`.
