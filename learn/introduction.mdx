## Zero-dependency drop-in

Drop in intelligence. No ollama daemon to keep alive. No LM Studio app running in the background. No Docker container. No HTTP boundary between your agent code and the model. `npm i`, point at a GGUF on disk, and you're orchestrating agents in the same Node process that runs your application.

<Tabs>
  <Tab title="lloyal HDK">
    ```typescript
    import { createContext } from "@lloyal-labs/lloyal.node";
    import { initAgents, agentPool, parallel } from "@lloyal-labs/lloyal-agents";

    const ctx = yield* call(() => createContext({ modelPath: "model.gguf" }));
    yield* initAgents(ctx);

    const pool = yield* agentPool({
      orchestrate: parallel(tasks),
      tools, terminalTool: 'report',
    });
    ```

    Single process. One install. The `Branch` your agent forks is the same object the inference loop mutates — no IPC, no JSON-over-HTTP, no model server protocol.
  </Tab>
  <Tab title="ollama">
    ```bash
    # Install daemon, keep it running
    brew install ollama
    ollama serve &
    ollama pull llama3
    ```

    ```typescript
    // HTTP-out from your code; KV state never visible
    const res = await fetch("http://localhost:11434/api/generate", { ... });
    ```

    Separate process. JSON over HTTP. Stateless from your code's perspective.
  </Tab>
  <Tab title="LM Studio">
    Desktop GUI to load and serve models. Local HTTP server speaks an OpenAI-compatible API. Same architectural boundary as ollama: separate process, no shared state with your agent code.
  </Tab>
  <Tab title="vLLM / TGI">
    Containerized GPU server. Production-grade throughput, but the deployment surface is a Docker image, a model checkpoint, and a runtime — not "hardware you own and edit on."
  </Tab>
</Tabs>

<Note>
  **Dependency footprint.** `@lloyal-labs/sdk` ships with zero production dependencies. `@lloyal-labs/lloyal-agents` adds [`effection`](https://frontside.com/effection/) (structured concurrency) and [`eta`](https://eta.js.org/) (templates). `@lloyal-labs/rig` adds `@mozilla/readability` and `linkedom` for web scraping. The `@lloyal-labs/lloyal.node` native binding is the only piece that touches the GPU.
</Note>

## Continuous Context

Most agent frameworks pass strings between stages. A transcript of what one agent generated becomes a prompt for the next; every step re-encodes the full context from text. The model's prior attention state — the specific weighted relationships it found between tokens — is discarded and rebuilt each call.

**lloyal agents share live attention state.** When stage 2 starts decoding, stage 1's findings are already in its KV at their original positions; its first generated token attends over them directly. No re-tokenization. No re-encoding. No "summarize, paste, re-prompt" information bottleneck.

<Tabs>
  <Tab title="Traditional agent loop">
    ```
    Stage 1 generates response
      ↓ extract findings as text
      ↓ build new prompt: system + tools + findings_string + query
      ↓ send to inference (re-tokenize, re-encode, re-attend over everything)
      ↓ Stage 2 attends over re-tokenized findings
    ```
    Re-encode every stage. Information serialized through text at every boundary.
  </Tab>
  <Tab title="Continuous Context loop">
    ```
    Stage 1 generates (decoded into KV positions 0..N)
      ↓ extendRoot prefills [user, asst] turn onto queryRoot
      ↓ Stage 2 forks queryRoot at the extended position (O(1) metadata)
      ↓ Stage 2's first token attends over positions 0..N+turn directly
    ```
    Fork is metadata-only. Findings are the same physical K/V pairs, not a re-encoding.
  </Tab>
</Tabs>

<Note>
  When the model computes Q·Kᵀ for its next token, it attends over all K vectors at positions 0..N — including those written during prior tool result prefills. Child branches share the same physical key-value pairs at inherited positions; not a re-encoding, the same vectors.
</Note>

A fork is a metadata operation on shared KV cells — O(1) cost, zero tensor copy. Three agents forked from the same root attend to the same physical key-value pairs for the prefix while writing their own unique context above the fork point.

### Why this breaks the "runs locally but is a toy" barrier

Re-encoding 900 tokens of system prompt and tool schemas per agent step is prohibitive on consumer hardware. Most local-LLM agent demos fall off a performance cliff at the second hop: three agents at 16K context each, each re-decoding the prefix, hits the wall in seconds.

With Continuous Context, that prefix is decoded once and inherited by every fork. Measured across a real pipeline: **4.4× fewer tokens processed** than a prompt-rebuilding approach. Same harness code runs on a 32K cloud GPU or a 16K laptop — depth adapts to the hardware.

### What emerges

Attention continuity changes what agents do with tools. Watching real pipelines, agents form and test hypotheses through iterative retrieval — narrowing, discovering, hypothesizing, then verifying — emergent from the attention mechanics, not from prompted instruction. Later searches reference concepts **absent from the original question**, discovered during earlier reads and still physically present in the KV cache.

## Retrieval Interleaved Generation

Classical RAG is a *pre-processing* step: embed the query, run a similarity search, concatenate top chunks into the prompt, generate. One shot, static context, no feedback loop between what the model reads and what it asks for next.

**RIG inverts this.** Retrieval happens *inside* the generation loop. The agent emits a tool call → a `Source` returns chunks → a cross-encoder reranker scores them against the original query → results are prefilled into the agent's KV at the call position → the next decoded token attends over them and decides what to ask for next. Hypothesis-driven, not bag-of-keywords.

<Tabs>
  <Tab title="Classical RAG">
    <Steps>
      <Step title="Embed query">
        Query passed through an embedding model → dense vector.
      </Step>
      <Step title="ANN search">
        Vector similarity search over the index → top-K chunks by cosine distance.
      </Step>
      <Step title="Concatenate">
        Build prompt: `system + retrieved_chunks + query`.
      </Step>
      <Step title="Generate">
        One inference call. Done.
      </Step>
    </Steps>
    Static context. The model never re-shapes retrieval based on what it learned.
  </Tab>
  <Tab title="Retrieval Interleaved Generation">
    <Steps>
      <Step title="Agent generates">
        PRODUCE phase samples a tool call (`web_search`, `fetch_page`, `search_corpus`, custom).
      </Step>
      <Step title="Source returns chunks">
        `Source<TCtx, TChunk>` queries the backend (web, corpus, vector store, database).
      </Step>
      <Step title="Reranker scores">
        Cross-encoder scores chunks against the original query. `shouldProceed` floor gate (default `0.25`) drops irrelevant.
      </Step>
      <Step title="Prefill at call position">
        SETTLE phase prefills the result into the agent's KV at the position where the call was made. Probe text optionally appended.
      </Step>
      <Step title="Attend and refine">
        Next PRODUCE attends over the prefilled result. The agent's next query is shaped by what it just read.
      </Step>
    </Steps>
    Dynamic. Each call is shaped by everything that came before in attention — including chunks the model has already cross-encoded against the original intent.
  </Tab>
</Tabs>

The kit ships:

<AccordionGroup>
  <Accordion title="Pluggable Source abstraction" icon="puzzle-piece">
    `Source<TCtx, TChunk>` is a small abstract base. Implement `name`, a `tools` getter, `bind(ctx)`, and `getChunks()`; inherit `createScorer(query)`. `WebSource` (Tavily-backed) and `CorpusSource` (filesystem-backed) ship with the kit; custom sources for vector stores, databases, JIRA, email, anything — all surface as Tools through the same path and all benefit from the rest of the RIG stack.
  </Accordion>
  <Accordion title="Cross-encoder reranker — four scoring roles" icon="layer-group">
    The reranker (Qwen3-Reranker-0.6B by default, auto-downloaded by the reference harness) drives an `EntailmentScorer` with four distinct methods:

    - **`scoreEntailmentBatch`** — texts vs. the original query. Boundary entailment for retrieved content.
    - **`scoreRelevanceBatch`** — dual-score `min(toolQueryScore, originalQueryScore)`. Used in exploit mode when KV pressure tightens focus.
    - **`scoreSimilarityBatch`** — texts vs. an arbitrary reference. Powers echo detection at delegation boundaries (the agent's own task as reference).
    - **`shouldProceed`** — floor gate. Default `_entailmentFloor = 0.25`.

    One model. Four roles. The reranker is RIG infrastructure, not a fetch optimization.
  </Accordion>
  <Accordion title="Dynamic prompt composition" icon="code">
    Eta templates parameterize on harness state. `web-worker.eta` adapts on `it.agentCount > 1` (sibling-task awareness for parallel mode) and `it.taskIndex > 0` (build-on-prior-research for chain mode). Synthesis prompts adapt on convergence state and source presence. Prompts are functions of the harness's current situation, not fixed strings.
  </Accordion>
  <Accordion title="Pressure-aware tool gating" icon="gauge">
    `DefaultAgentPolicy` denies non-terminal tool calls when KV is over budget. The terminal tool (`report`) always passes — the agent can finish even under pressure. The explore/exploit gate (`_exploreContext = 0.4` by default) flips the reranker's chunk-filtering mode when ≤40% of context remains. Retrieval is bounded by the available compute, not by guesswork.
  </Accordion>
  <Accordion title="Default delegation guards" icon="shield-halved">
    Out of the box: research-before-delegate (`_minToolCalls = 2` — agents must do at least one search and one read before delegating), fetch-URL dedup, search-query dedup, and `echoThreshold = 0.8` for paraphrased recursion. Production failure modes prevented at the framework layer, not at every harness author's expense.
  </Accordion>
  <Accordion title="Continuous Context across delegation" icon="diagram-project">
    `DelegateTool`'s warm path forks sub-agents from `parent: context.branch`. The sub-agent inherits the calling agent's full KV state — system prompt, tool schemas, prior search results, prior page reads — at zero marginal cost. The deeper an investigation goes, the richer the attention state at depth.
  </Accordion>
</AccordionGroup>

## The stack

```
Your harness
    ↓
@lloyal-labs/rig              Tools, sources, RIG building blocks
    ↓
@lloyal-labs/lloyal-agents    Continuous Context agent orchestration
    ↓
@lloyal-labs/sdk              Inference runtime: branches, sessions, KV cache
    ↓
@lloyal-labs/lloyal.node      Native GPU backend (macOS · Linux · Windows)
```

- **[Tools](/learn/tools)** — anything an agent can call: databases, APIs, filesystems, web search, services. You define the interface; the model dispatches uniformly.
- **[Agent Pools](/learn/agents)** — parallel agents on shared KV. Orchestrators (`parallel` / `chain` / `fanout` / `dag`) reshape the harness without changing the call.
- **[Sources](/learn/sources)** — any data backend: web, corpus, vector store, database. Five-line contract; full RIG stack inherited.
- **[Pipelines](/learn/pipelines)** — compose pool stages into research → synth → verify workflows.
