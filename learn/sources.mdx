A source is any system your agents can search. File system. Web. Vector database. Email inbox. JIRA. Internal wiki. Anything with content and a way to find it.

You define how to search. The framework handles agent orchestration, tool dispatch, and result collection.

## The contract

Every source implements five members:

```typescript
import { Source } from "@lloyal-labs/lloyal-agents";
import type { Tool } from "@lloyal-labs/lloyal-agents";
import type { Operation } from "effection";

class MySource extends Source<MyContext, MyChunk> {
  // 1. Human-readable label for this source
  readonly name = "jira";

  // 2. The research tool — an atomic swarm that searches this source
  get researchTool(): Tool {
    return this._researchTool;
  }

  // 3. Late-bind runtime dependencies not available at construction
  *bind(ctx: MyContext): Operation<void> {
    this._reranker = ctx.reranker;
    this._researchTool = new JiraResearchTool(this._reranker);
  }

  // 4. Post-research chunks for reranking
  getChunks(): MyChunk[] {
    return this._buffer.getAll();
  }

  // 5. Grounding tools for independent verification
  get groundingTools(): Tool[] {
    return [new JiraSearchTool(this._client)];
  }
}
```

<Steps>
  <Step title="name">
    A label. Used in findings to attribute which source produced what. Keep it short — `"web"`, `"corpus"`, `"jira"`.
  </Step>

  <Step title="researchTool">
    The main tool. When called, it spawns agents that search this source and return findings. This is typically a recursive tool — it creates a `withSharedRoot` + `runAgents` internally.

    The orchestrator doesn't know how your source works. It calls `researchTool.execute({ questions })` and gets back findings.
  </Step>

  <Step title="bind(ctx)">
    Called before tools are used. Wire up runtime dependencies — reranker, API clients, database connections — that weren't available at construction time. This is a generator, so you can do async setup.
  </Step>

  <Step title="getChunks()">
    Returns buffered content for post-research reranking. During research, agents fetch content (pages, documents, code files). The source buffers it. After research completes, the orchestrator calls `getChunks()` and reranks against the original query to select the best passages for synthesis.
  </Step>

  <Step title="groundingTools">
    Tools for independent verification. When research agents produce conflicting findings, the synthesis agent receives these tools so it can read the source directly to resolve contradictions — without running a full research swarm. When findings converge, grounding tools are not needed.
  </Step>
</Steps>

## How sources compose

A pipeline can use multiple sources. The orchestrator iterates through them:

```typescript
const sources = [
  new CorpusSource(resources),  // Search local files first
  new WebSource(tavilyKey),     // Then search the web
];

// In the pipeline:
for (const source of sources) {
  yield* source.bind(ctx);
  const result = yield* source.researchTool.execute({ questions });
  findings.push(result);
}
```

Each source is independent. It builds its own tools, manages its own agents, and returns its own findings. The orchestrator sees a uniform interface.

<Tip>
Sources are ordered deliberately. Put high-confidence sources first (local corpus, verified databases). Web or broad API sources go last. Between sources, a bridge stage can structure early findings as context for later sources.
</Tip>

## Building your own source

Start with the question: what can you search, and what do you get back?

```typescript
class SlackSource extends Source<SourceContext, SlackChunk> {
  readonly name = "slack";
  private _client: SlackClient;
  private _buffer: SlackChunk[] = [];
  private _researchTool!: Tool;

  constructor(token: string) {
    super();
    this._client = new SlackClient(token);
  }

  get researchTool(): Tool { return this._researchTool; }

  *bind(ctx: SourceContext): Operation<void> {
    // Build a tool that searches Slack channels and threads
    this._researchTool = new SlackResearchTool(
      this._client, this._buffer, ctx.reranker
    );
  }

  getChunks(): SlackChunk[] { return this._buffer; }

  get groundingTools(): Tool[] {
    return [new SlackSearchTool(this._client)];
  }
}
```

The research tool is where the work happens. It's a regular `Tool` subclass that spawns agents:

```typescript
class SlackResearchTool extends Tool<{ questions: string[] }> {
  readonly name = "slack_research";
  // ...

  *execute(args: { questions: string[] }, context?: ToolContext): Operation<unknown> {
    const pool = yield* withSharedRoot(
      { systemPrompt: SLACK_PROMPT, tools: toolsJson, parent: context?.branch },
      //                                               ^^^^^^^^^^^^^^^^^^^^^^
      // Warm path: fork from calling agent's branch (inherits full attention state)
      function*(root) {
        return yield* runAgents({
          tasks: args.questions.map(q => ({
            systemPrompt: SLACK_PROMPT,
            content: q,
            tools: toolsJson,
            parent: root,
          })),
          tools: toolMap,
          terminalTool: "report",
          pruneOnReport: true,    // free KV immediately when agents report
          reportPrompt: REPORT,   // extract findings from hard-cut agents
        });
      },
    );
    return {
      findings: pool.agents.map(a => a.findings).filter(Boolean),
      supportingFindings: pool.agents.flatMap(a => a.childFindings ?? []),
    };
  }
}
```

<Note>
The source abstraction is intentionally thin. Five members. No configuration framework, no registration system, no plugin lifecycle. You subclass `Source`, implement the contract, and pass instances to your pipeline.
</Note>
