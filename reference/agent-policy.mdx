Every agent lifecycle decision passes through a policy. When an agent hits a stop token, when a tool result won't fit in KV, when a tool call is about to dispatch — the pool consults the policy and executes the returned action. The pool handles mechanics (transitions, prefills, prunes). The policy handles judgment.

## The interface

```typescript
interface AgentPolicy {
  onProduced(agent, parsed, pressure, config): ProduceAction;
  onSettleReject(agent, resultTokens, pressure, config): SettleAction; // deprecated
  shouldExplore?(agent, pressure): boolean;
  shouldExit?(agent, pressure): boolean;
  onRecovery?(agent): RecoveryAction;
  readonly pressureThresholds?: PressureThresholds;
  resetTick?(): void;
}
```

Five hooks plus a getter, called at different points in the [tick loop](/reference/concurrency#four-phase-tick-loop):

| Hook | Phase | When | What it decides |
|------|-------|------|-----------------|
| `shouldExit` | PRODUCE (pre-generate) | Before each agent produces tokens | Kill immediately — no chance to respond |
| `onProduced` | PRODUCE (post-generate) | Agent hit stop token | Tool call, report, nudge, or kill |
| `onSettleReject` | SETTLE | Tool result exceeds headroom | Nudge or kill |
| `shouldExplore` | DISPATCH | Before each tool executes | Explore mode or exploit mode |
| `onRecovery` | Post-loop | For each killed agent without result | Extract findings or skip |
| `pressureThresholds` | Pool setup | Once at pool initialization | KV softLimit and hardLimit |

The policy receives the agent's full state — status, tool history, result, token count — plus a live [ContextPressure](/reference/kv-pressure) snapshot and pool configuration. It returns an action. The pool executes it.

## `shouldExit` — pre-generate kill gate

Called before `produceSync()` — before the agent generates any tokens. Returns `true` to kill the agent immediately. There's no nudge possible here — there's no tool call to attach an error message to. The branch stays alive for recovery via `onRecovery`.

```typescript
// PRODUCE: before each agent generates
if (policy.shouldExit?.(agent, pressure) ?? pressure.critical) {
  // transition to idle, eligible for recovery
}
```

Optional — defaults to `pressure.critical` when absent. `DefaultAgentPolicy` also checks the time budget:

- `pressure.critical` (remaining < hardLimit) → exit
- `budget.time.hardLimit` exceeded → exit

## `onProduced` — the main decision point

Called after `parseChatOutput` extracts content and tool calls from the agent's generation. The policy sees what the agent wants to do and decides what actually happens.

Returns one of five actions:

| Action | What the pool does |
|--------|-------------------|
| `tool_call` | Dispatch the tool in DISPATCH phase |
| `report` | Extract result, mark agent idle, emit `agent:report` |
| `nudge` | Inject an error message as a fake tool result |
| `idle` | Kill the agent — transition to idle, emit `agent:done` |
| `free_text_report` | Agent produced prose without a tool call — capture as result |

### Decision flow in `DefaultAgentPolicy`

```
parsed output has tool call?
  ├─ No → free_text_report (if agent has done work) or idle
  │
  ├─ Under pressure? (turns >= maxTurns OR headroom < 0 OR time softLimit)
  │   ├─ Non-terminal tool → nudge (stateless, every time)
  │   └─ Critical pressure or no terminal tool → idle (kill)
  │
  ├─ Terminal tool? (e.g. report)
  │   ├─ Not under pressure + < minToolCalls → nudge ("use tools first")
  │   └─ Under pressure OR sufficient work → report
  │
  ├─ Guard rejected? → nudge (with guard's message)
  │
  └─ All checks pass → tool_call
```

The terminal tool is never dispatched — the policy intercepts it, extracts the `result` field from the tool arguments, and returns `report`. When the agent is under pressure, early reports are accepted regardless of tool call count — an agent told to "report now" should be allowed to report.

## `onSettleReject` — deprecated

<Note>
`onSettleReject` is retained for backward compatibility with custom policies but is no longer called by the pool. SETTLE now defers oversized tool results until headroom recovers via the trailing stop in PRODUCE. See [KV Pressure: SETTLE deferral](/reference/kv-pressure#4-settle-deferral-tool-result-gating).
</Note>

The method signature is unchanged. Custom policies that override it will compile but the override has no effect.

## `shouldExplore` — explore vs exploit mode

Called at DISPATCH before each tool executes. Returns `true` for explore mode (agent-local scoring at content boundaries) or `false` for exploit mode (dual scoring via [scoreRelevanceBatch](/reference/search-strategy#content-boundaries-explore-vs-exploit)).

Optional — defaults to `true` (explore) when absent. `DefaultAgentPolicy` switches based on `pressure.percentAvailable`:

- Above `exploreThreshold` (default 40) → explore
- Below → exploit

<Note>
`percentAvailable` is live — it recovers when branches are pruned. An agent in exploit mode can flip back to explore if other agents report and free KV. This is intentional. Explore/exploit is a non-monotonic quality knob, separate from the lifecycle (nudge/kill/recover). See [KV Pressure — cellsUsed tracks live pressure](/reference/kv-pressure#cellsused-tracks-live-pressure).
</Note>

## The nudge mechanism

Nudging is stateless — no escalation, no tracking of past nudges. When the agent is over budget and calls a non-terminal tool, the policy returns `nudge` with a message. The pool:

1. Serializes the message as a tool result: `{ "error": "..." }`
2. Prefills the error into the agent's branch
3. Resets the grammar so the agent generates a fresh response

The agent sees the error in its context and may call the terminal tool on its next turn. If it tries another non-terminal tool while still over budget, the policy nudges again. If headroom recovers (branches pruned, KV freed), the agent is no longer over budget and can continue working.

There is no "second offense" kill. The hard kill is `shouldExit`'s job — triggered by `pressure.critical` or time `hardLimit`, not by repeated nudges.

Three sources of nudge messages:

| Source | Message |
|--------|---------|
| KV pressure | "KV memory pressure — report your findings now." |
| Turn limit | "Turn limit reached — report your findings now." |
| Time budget | "Time limit approaching — report your findings now." |
| Insufficient work | "You must use tools before submitting results." |
| Tool guard | The guard's custom `message` field |
## `resetTick` — per-tick state reset

Called by the pool at the start of each tick iteration. Resets per-tick flags used by the trailing stop:

- `_killedThisTick`: At most one agent killed per critical tick
- `_nudgedThisTick`: At most one agent nudged per over-budget tick

Optional — only needed if the policy tracks per-tick state. `DefaultAgentPolicy` implements this.

## `onRecovery` — scratchpad extraction

Called inline when the trailing stop kills an agent during PRODUCE. The policy decides whether to extract findings from the agent's branch via grammar-constrained generation.

```typescript
type RecoveryAction =
  | { type: 'extract'; prompt: { system: string; user: string } }
  | { type: 'skip' };
```

`DefaultAgentPolicy` checks the confabulation guard before returning `extract`:
- `agent.tokenCount >= minTokens` (default 100) — agent generated enough context
- `agent.toolCallCount >= minToolCalls` (default 2) — agent did real work

Agents that barely ran would produce fabricated findings — the guard prevents this.

The pool owns the extraction grammar (`{ "result": "..." }` JSON schema). The policy owns the prompt and the decision.

<Tip>
Recovery runs after all agents finish, using KV freed by agents that already reported. Set `pruneOnReport: true` to maximize headroom for extraction passes.
</Tip>

## Tool guards

Guards are declarative rejection rules checked inside `onProduced` before returning `tool_call`. Each guard specifies which tools it applies to and a synchronous reject function:

```typescript
interface ToolGuard {
  tools: string[];
  reject: (args, lineageHistory, agent) => boolean;
  message: string;
}
```

`DefaultAgentPolicy` ships three guards:

### Fetch deduplication

Rejects `fetch_page` if the same URL was already fetched in the agent's lineage (self + ancestors). Uses `walkAncestors` to collect tool history across the full delegation chain.

### Search deduplication

Rejects `web_search` if the same query (case-insensitive) was already searched in the lineage.

### Research-before-delegation

Rejects `web_research` / `research` unless the agent has done at least one search AND one fetch/read **in its own local history** — not lineage.

<Note>
This guard deliberately uses agent-local history, not lineage. With lineage history, a child agent would inherit its parent's search and fetch records, letting it call the delegation tool on turn 0 without doing any work of its own. The local-history check forces each agent to investigate before delegating.
</Note>

### Custom guards

Replace the defaults entirely or append additional guards:

```typescript
const policy = new DefaultAgentPolicy({
  guards: [myGuard],           // replaces all defaults
  // OR
  extraGuards: [myGuard],      // appends to defaults
});
```

## `DefaultAgentPolicy` configuration

```typescript
import { DefaultAgentPolicy } from "@lloyal-labs/lloyal-agents";

const policy = new DefaultAgentPolicy({
  // Tool guards
  minToolCallsBeforeReport: 2,  // default 2
  extraGuards: [],

  // Search strategy (quality knob — non-monotonic, flips with pressure)
  exploreThreshold: 40,          // default 40

  // Budget (lifecycle knob — nudge at softLimit, kill at hardLimit)
  budget: {
    context: { softLimit: 1024, hardLimit: 128 },  // KV tokens remaining
    time: { softLimit: 480_000, hardLimit: 600_000 },  // ms since policy creation
  },

  // Recovery (scratchpad extraction for killed agents)
  recovery: {
    prompt: { system: "Summarize all findings.", user: "Report your findings." },
    minTokens: 100,      // confabulation guard
    minToolCalls: 2,
  },
});
```

| Option | Default | Effect |
|--------|---------|--------|
| `minToolCallsBeforeReport` | 2 | Agents with fewer non-terminal tool calls get nudged when trying to report (unless under pressure) |
| `guards` | 3 default guards | Replaces the entire guard array when provided |
| `extraGuards` | `[]` | Appended to default guards |
| `exploreThreshold` | 40 | KV percent available threshold for explore/exploit switch |
| `budget.context.softLimit` | 1024 | KV tokens remaining that triggers nudge |
| `budget.context.hardLimit` | 128 | KV tokens remaining that triggers kill via `shouldExit` |
| `budget.time.softLimit` | — | Milliseconds since policy creation that triggers nudge |
| `budget.time.hardLimit` | — | Milliseconds since policy creation that triggers kill |
| `recovery.prompt` | — | System + user prompt for scratchpad extraction |
| `recovery.minTokens` | 100 | Skip extraction for agents with fewer tokens |
| `recovery.minToolCalls` | 2 | Skip extraction for agents with fewer tool calls |

### `pressureThresholds`

The pool reads `policy.pressureThresholds` once at setup to configure `ContextPressure` snapshots. Returns `{ softLimit, hardLimit }` from `budget.context` or the class defaults (1024, 128).

### `setExploitMode`

Harness-driven override for immediate exploit mode:

```typescript
policy.setExploitMode(true);   // all agents exploit regardless of pressure
policy.setExploitMode(false);  // revert to pressure-driven
```

<Note>
`setExploitMode` is on `DefaultAgentPolicy` (the class), not `AgentPolicy` (the interface). Not all policy implementations support forced exploit. Harness code that needs it must type the variable as `DefaultAgentPolicy`.
</Note>

### Time budget scope

The `_startTime` is set when the policy is constructed. Since the same policy instance is shared across all nesting levels (outer pool and inner pools spawned by DelegateTool), the time budget is global — "the research has a 10 minute budget" applies to all agents at every depth.

Create separate policy instances for stages with different time constraints:

```typescript
const researchPolicy = new DefaultAgentPolicy({
  budget: { context: { softLimit: 1024 }, time: { softLimit: 480_000, hardLimit: 600_000 } },
  recovery: { prompt: REPORT },
});

// Synthesis doesn't need time limits — use default policy
yield* createAgentPool({ tasks, systemPrompt, tools, policy: researchPolicy, ... });
```

## Writing a custom policy

Implement the `AgentPolicy` interface to control any lifecycle decision:

```typescript
import type { AgentPolicy, ProduceAction, SettleAction, RecoveryAction } from "@lloyal-labs/lloyal-agents";
import type { ContextPressure } from "@lloyal-labs/lloyal-agents";

class MyPolicy implements AgentPolicy {
  onProduced(agent, parsed, pressure, config): ProduceAction {
    // Return: tool_call, report, nudge, idle, or free_text_report
  }

  onSettleReject(agent, resultTokens, pressure, config): SettleAction {
    // Return: nudge (with message) or idle
  }

  shouldExplore(agent, pressure): boolean {
    return pressure.percentAvailable > 50;
  }

  shouldExit(agent, pressure): boolean {
    return pressure.critical;
  }

  onRecovery(agent): RecoveryAction {
    if (agent.tokenCount < 100) return { type: 'skip' };
    return { type: 'extract', prompt: { system: '...', user: '...' } };
  }

  get pressureThresholds() {
    return { softLimit: 2048, hardLimit: 256 };
  }
}
```

Pass to any pool via the `policy` option:

```typescript
yield* createAgentPool({ tasks, tools, systemPrompt, policy: new MyPolicy() });
```

Signatures are narrow by design: `(agent, pressure)`. The policy is a class — time, cost, or other signals live on `this`. The pool passes what it owns (agent + pressure snapshot); the policy combines with its own state.

The policy instance is shared across all agents in a pool. For nested pools (recursive tools), the inner pool receives the same policy instance.
