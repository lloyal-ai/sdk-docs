Three agents. One GPU. Shared prefix.

```
Shared root (system prompt + tool schemas): decoded once
  ‚îú‚îÄ‚îÄ Agent 0: fork ‚Üí investigate question A
  ‚îú‚îÄ‚îÄ Agent 1: fork ‚Üí investigate question B
  ‚îî‚îÄ‚îÄ Agent 2: fork ‚Üí investigate question C
```

Every agent inherits the root's KV cache at zero cost ‚Äî an O(1) metadata tag, not a copy. Each agent then generates independently, calls tools, and reports findings. The pool manages the entire lifecycle.

## What is an Agent?

An agent is a **branch with intent**. A Branch is a forkable KV cache sequence ‚Äî it stores every token the model has seen and generated. An Agent adds: a task to accomplish, a policy for how to accomplish it, and a record of what it has done.

The branch is the ground truth. The agent is the interpretation layer that gives meaning to what's in the branch and makes decisions about what to do next.

```typescript
import { Agent } from "@lloyal-labs/lloyal-agents";

// The pool creates agents for you ‚Äî you don't instantiate them directly.
// After the pool completes, each AgentResult exposes:
pool.agents[0].result       // what the agent discovered (or null)
pool.agents[0].toolCallCount  // how many tools it called
pool.agents[0].tokenCount     // how many tokens it generated
pool.agents[0].nestedResults  // results from recursive sub-agents
pool.agents[0].ppl            // model-level perplexity at completion
```

### Status model

Every agent transitions through a lifecycle:

```
idle ‚Üí active ‚Üí awaiting_tool ‚Üí active ‚Üí ... ‚Üí idle ‚Üí disposed
```

- **idle** ‚Äî created but not yet generating, OR finished but branch still alive (extraction window)
- **active** ‚Äî generating tokens between stop boundaries
- **awaiting_tool** ‚Äî tool call parsed, waiting for result in SETTLE
- **disposed** ‚Äî branch pruned, agent no longer usable

The pool enforces valid transitions. An agent cannot go backward or skip states.

### Findings

Agents produce results through a single write path with provenance tracking:

- `report_tool` ‚Äî agent voluntarily called `report()`
- `free_text` ‚Äî agent emitted prose without a tool call
- `scratchpad` ‚Äî extracted post-idle via grammar-constrained fork
- `tool_error` ‚Äî tool threw, error captured as result

### Tool history

Each agent records metadata for every tool invocation ‚Äî tool name, summarized arguments, result token count, context available after settlement. The [Agent Policy](#agent-policy) reads this history for informed lifecycle decisions.

### Parent lineage

When a tool spawns sub-agents (recursive tools like `research` or `web_research`), each sub-agent records which agent called the tool that created it. `walkAncestors()` traverses the chain ‚Äî self ‚Üí caller ‚Üí caller's caller ‚Äî collecting data from each ancestor. Tool guards use this for cross-level deduplication.

## The pattern

Two functions compose to create a parallel agent pool:

```typescript
import { withSharedRoot, runAgents, createToolkit } from "@lloyal-labs/lloyal-agents";

const { toolMap, toolsJson } = createToolkit([...tools, reportTool]);

const pool = yield* withSharedRoot(
  { systemPrompt: RESEARCH_PROMPT, tools: toolsJson },
  function*(root) {
    return yield* runAgents({
      tasks: questions.map(q => ({
        systemPrompt: RESEARCH_PROMPT,
        content: q,
        tools: toolsJson,
        parent: root,
      })),
      tools: toolMap,
      terminalTool: "report",
      maxTurns: 6,
      policy: researchPolicy,    // budget + recovery configured on policy
      pruneOnReport: true,       // free KV immediately when agents report
    });
  },
);

// pool.agents[0].result, pool.agents[1].result, ...
```

`withSharedRoot` creates the root branch, prefills the system prompt, and passes the root to your body function. When the body returns, the root and all its children are pruned automatically.

`runAgents` forks one branch per task from the root, runs them in parallel, and returns when all agents finish. Branches are pruned on exit.

### `recovery` ‚Äî scratchpad extraction

When agents are killed by KV pressure before calling `report()`, the policy can extract partial findings from their accumulated context. Configure `recovery` on `DefaultAgentPolicyOpts` with a prompt ‚Äî the policy's `onRecovery()` decides per-agent whether to extract, and the pool forks from the agent's branch with a grammar-constrained schema.

A confabulation guard skips agents that barely ran. The thresholds are configurable:

```typescript
const researchPolicy = new DefaultAgentPolicy({
  recovery: {
    prompt: { system: "You are a research reporter.", user: "Report your findings." },
    minTokens: 100,     // skip extraction below this (default 100)
    minToolCalls: 2,     // skip extraction below this (default 2)
  },
});
```

An agent with insufficient context would produce hallucinated findings ‚Äî the guard prevents this.

### `pruneOnReport`

When `true`, agents are pruned immediately after reporting. This frees KV mid-pool, giving remaining agents more headroom. Without it, all branches stay alive until the pool completes.

## Agent Policy

The pool consults an **AgentPolicy** at every decision boundary ‚Äî when an agent hits a stop token, when a tool result doesn't fit in KV, and when checking tool guards before dispatch.

```typescript
interface AgentPolicy {
  onProduced(agent, parsed, pressure, config): ProduceAction;
  onSettleReject(agent, resultTokens, pressure, config): SettleAction;
  shouldExplore?(agent, pressure): boolean;  // optional ‚Äî defaults to true
}
```

`DefaultAgentPolicy` handles all the default behavior:

- **Terminal tool interception** ‚Äî when the agent calls `report()`, extract result and mark idle
- **Budget enforcement** ‚Äî deny non-terminal tool calls when over budget, nudge the agent to report
- **Confabulation prevention** ‚Äî reject premature reports from agents with fewer than 2 tool calls
- **Tool guard checking** ‚Äî run declarative guards before dispatching any tool call
- **Explore/exploit switching** ‚Äî decide per-DISPATCH whether content-boundary tools should explore (agent-local scoring) or exploit (dual scoring against original query)

Customize thresholds and guards via the constructor:

```typescript
import { DefaultAgentPolicy } from "@lloyal-labs/lloyal-agents";

const policy = new DefaultAgentPolicy({
  minToolCallsBeforeReport: 3,   // require 3 tool calls before report (default 2)
  extraGuards: [myCustomGuard],  // append to default guards
  exploreThreshold: 40,          // switch to exploit when KV < 40% available (default 40)
});

yield* runAgents({
  tasks, tools: toolMap,
  policy,                         // inject custom policy
});
```

Or replace the default guards entirely with `guards: [...]`.

The policy sees the agent's full state: status, tool history, result, nudge count, pressure snapshot. It returns an action ‚Äî `tool_call`, `report`, `nudge`, or `idle` ‚Äî and the pool executes it.

#### Explore vs exploit mode

`shouldExplore` is evaluated at every DISPATCH with a fresh `ContextPressure` snapshot (post-SETTLE, reflecting actual KV usage). When it returns `true` (explore), content-boundary tools (search, fetch_page) use agent-local reranker scoring only ‚Äî preserving serendipitous discovery of bridging content. When it returns `false` (exploit), tools apply `scoreRelevanceBatch` ‚Äî dual scoring against both the tool's query and the original root query via `min()` ‚Äî tightening focus at the cost of discovery.

`DefaultAgentPolicy` switches based on `pressure.percentAvailable`: above `exploreThreshold` (default 40) ‚Üí explore, below ‚Üí exploit. The harness can override with `policy.setExploitMode(true)` to force exploit mode immediately (e.g. when the user wants agents to wrap up).

<Note>
`setExploitMode` is on `DefaultAgentPolicy` (the class), not `AgentPolicy` (the interface). Harness code that needs it must type the variable as `DefaultAgentPolicy`, not `AgentPolicy`.
</Note>

### Tool Guards

Guards are declarative rejection rules checked before tool dispatch. Each guard specifies which tools it applies to and a reject function:

```typescript
interface ToolGuard {
  tools: string[];
  reject: (args, lineageHistory, agent) => boolean;
  message: string;
}
```

Four checks guard the delegation boundary:

1. **Fetch dedup** ‚Äî rejects `fetch_page` if the same URL was already fetched by the agent or any ancestor. Uses lineage history.
2. **Search dedup** ‚Äî rejects `web_search` if the same query (case-insensitive) was already searched in the lineage. Uses lineage history.
3. **Research-before-delegation** ‚Äî rejects `web_research` / `research` unless the agent has done at least one search AND one fetch/read itself. Uses **agent-local** history, not lineage.
4. **Echo detection** ‚Äî rejects delegation if ALL proposed sub-questions are paraphrases of the agent's own task. Scores proposed questions against `Agent.task` via `scoreSimilarityBatch` (async, runs inside the delegation tool rather than the synchronous ToolGuard interface). If `min(scores) > echoThreshold` (default 0.8), the delegation is pure repetition ‚Äî rejected. If ANY question scores below the threshold, there's a novel thread and the delegation proceeds.

Guards 1-3 are synchronous `ToolGuard` instances on `DefaultAgentPolicy`. Guard 4 uses the reranker (async) and runs in the delegation tool alongside the [entailment gate](#entailment-scoring). Configure the echo threshold via `spawnAgents`:

```typescript
yield* spawnAgents({
  // ...
  echoThreshold: 0.8,  // reject if min(similarity to own task) > 0.8
});
```

<Note>
The research-before-delegation guard (3) uses agent-local history deliberately. With lineage history, a child agent would inherit its parent's search and fetch records, letting it call `web_research` on turn 0 without doing any work. The echo detection guard (4) catches the next failure mode: agents that DO research but then delegate verbatim copies of their own task. Because sub-agents inherit the full parent prefix via Continuous Context, echoed delegation produces children that search the same queries, find the same results, and delegate again ‚Äî compounding waste at each depth.
</Note>

## The four-phase tick loop

All agents generate tokens together in a batched loop. Each tick has four phases:

<Steps>
  <Step title="PRODUCE">
    Every active agent samples one token via `produceSync()`. If the token is a stop token, the pool parses the output for tool calls or final content. The policy decides what to do ‚Äî dispatch the tool, report findings, nudge the agent, or mark it idle. No async gaps ‚Äî pure synchronous sampling.
  </Step>

  <Step title="COMMIT">
    A single GPU call via `store.commit()` batch-decodes all produced tokens. One `llama_decode` for the entire pool, not one per agent.
  </Step>

  <Step title="SETTLE">
    Tool results from the previous tick's dispatch are drained. Each result is tokenized and prefilled into the calling agent's branch. Results that exceed available headroom are rejected ‚Äî the policy decides whether to nudge the agent or kill it. Agents resume generating with the tool result in their attention state.
  </Step>

  <Step title="DISPATCH">
    Collected tool calls execute sequentially ‚Äî one at a time. Each tool runs inside a `scoped()` error boundary. The pool sets `CallingAgent` context before each dispatch so recursive tools can identify their caller. When a tool returns result arrays (from recursive sub-agents), the pool collects them as `nestedResults` on the calling agent.
  </Step>
</Steps>

<Note>
Tools execute sequentially because they may access the GPU (recursive tools spawn inner pools). Concurrent tool execution would cause concurrent `llama_decode` calls on the same context ‚Äî which crashes. The pool serializes dispatch to guarantee exclusive GPU access.
</Note>

## Prefix sharing

The shared root is the key efficiency mechanism. System prompt + tool schemas are decoded into KV once. Every agent fork inherits them.

```
Without sharing: 3 agents x 900 tokens = 2,700 GPU decode ops
With sharing:    900 shared + 3 x 30 unique = 990 GPU decode ops
Savings:         64% fewer decode operations
```

In real pipelines with multi-turn agents, the savings compound further ‚Äî tool results stay in KV as deltas, never re-sent. Measured savings reach **78% fewer tokens** across a full research run.

### Warm path for recursive tools

When a tool spawns sub-agents, `withSharedRoot` can fork from the calling agent's branch instead of creating a new root at position 0. The calling agent already has the system prompt in its KV, so the sub-agent pool only needs to prefill the turn separator ‚Äî **2 tokens instead of ~760**.

```typescript
// Inside a tool's execute() method:
yield* withSharedRoot(
  { systemPrompt, tools: toolsJson, parent: context.branch },
  //                                 ^^^^^^^^^^^^^^^^^^^^^^
  // Forks from the calling agent's branch ‚Äî inherits full attention state
  function*(root) {
    return yield* useAgentPool({ tasks, tools: toolMap, ... });
  },
);
```

Sub-agents inherit every tool call, every tool result, every token the parent generated. The sub-agent's first token attends over the parent's full accumulated state. This is [Continuous Context](/learn/introduction#continuous-context-agents-agents-branch-inherit-and-build-on-each-others-attention-state).

<Tip>
For the full picture ‚Äî KV layout, position-aware decode, nested sharing, measured numbers ‚Äî see [Prefix Sharing](/reference/prefix-sharing).
</Tip>

## Nested results

When agents delegate to sub-agents (via `spawnAgents` with `recursive: true`), inner results can get lost. The outer agent sees the inner results as text in its KV, but when it reports, it may cherry-pick one and drop the rest.

`nestedResults` solves this. During DISPATCH, the pool collects result arrays from recursive tool returns and stores them on the calling agent. These bubble up through `AgentResult`:

```typescript
const pool = yield* spawnAgents({ tools, systemPrompt, tasks, ... });

// Primary findings ‚Äî what each agent reported
pool.agents[0].result          // "The DOJ connects iPod to..."

// Nested results ‚Äî what inner sub-agents found (flattened from recursive tools)
pool.agents[0].nestedResults     // ["Sub-agent found X", "Sub-agent found Y"]
```

Any tool that returns `{ results: string[] }` or `{ nestedResults: string[] }` automatically participates ‚Äî no tool-name hardcoding. The collection is tool-agnostic.

## Entailment scoring

When agents recurse, their local hypotheses can drift from the original query. An agent reads about "speculative execution in Apple Silicon" (CPU architecture), extracts domain terms, and delegates about them ‚Äî following the wrong thread for 4 levels. The model can't distinguish lexical similarity from semantic relevance.

The reranker (a 0.6B cross-encoder with its own context, zero inference KV cost) acts as a real-time entailment oracle. The `EntailmentScorer` interface provides three scoring methods, each operating against a different query:

| Method | Scores against | Boundary type | Purpose |
|--------|---------------|---------------|---------|
| `scoreEntailmentBatch(texts)` | **Original query** (closure) | Steering | Filter drifted search results and delegation questions |
| `scoreRelevanceBatch(texts, localQuery)` | **min(tool query, original query)** | Content (exploit mode only) | Tighten focus when KV headroom is low |
| `scoreSimilarityBatch(reference, texts)` | **Arbitrary reference** (e.g. agent task) | Steering | Echo detection ‚Äî reject paraphrased delegations |

**Steering boundaries** (always scored against original query):
- **Web search results** ‚Äî reranked by `scoreEntailmentBatch` before the agent sees them
- **Delegation boundary** ‚Äî proposed sub-questions filtered by `scoreEntailmentBatch`, then checked for echo via `scoreSimilarityBatch`

**Content boundaries** (agent-local by default, dual-scored in exploit mode):
- **Explore mode** (default): fetch_page chunks and corpus search results are scored against the **agent's local query only**. This preserves serendipitous discovery ‚Äî adjacent content that the agent didn't ask for but that connects the investigation to the answer.
- **Exploit mode** (low KV headroom): `scoreRelevanceBatch` applies `min(toolQueryScore, originalQueryScore)` ‚Äî demoting content that's locally relevant but drifted from the root intent. Controlled by [explore/exploit mode](#explore-vs-exploit-mode).

```typescript
// Source creates an immutable scorer per research invocation
const scorer = source.createScorer(query);

const pool = yield* spawnAgents({
  tools: source.tools,
  systemPrompt: RESEARCH_PROMPT,
  tasks: questions,
  scorer,   // propagated to all depths via ToolContext
  // ...
});
```

The scorer propagates through the entire recursion tree ‚Äî every DelegateTool reads `context.scorer` and passes it to the inner `spawnAgents()` call. The same immutable scorer reaches every depth.

<Note>
`createScorer()` captures the original query in a closure. No mutable state on Source. Safe to use across concurrent pools within the same research run. Different queries get different scorers. The three-query distinction (tool query, agent task, original query) is critical ‚Äî see the JSDoc on `EntailmentScorer` in source.ts for the canonical reference.
</Note>

## Context pressure

The pool monitors KV cache usage every tick and enforces two thresholds:

- **Soft limit** (default 1024 tokens) ‚Äî headroom reserved for downstream work. When exhausted, non-terminal tool calls are rejected. Terminal tools (like `report`) always pass through.
- **Hard limit** (default 128 tokens) ‚Äî crash-prevention floor. Agents are killed before `produceSync()` to prevent `llama_decode` failures.

You can tune these via the policy's `budget.context`:

```typescript
const policy = new DefaultAgentPolicy({
  budget: { context: { softLimit: 2048, hardLimit: 256 } },
});

yield* runAgents({ tasks, tools: toolMap, policy });
```

Tool results include `_contextAvailablePercent` ‚Äî derived from `ContextPressure.percentAvailable` (0‚Äì100 integer). Agents see this in their attention window, enabling pressure-aware decisions. The same percentage drives [explore/exploit switching](#explore-vs-exploit-mode) via `policy.shouldExplore()`.

<Note>
Pressure management is automatic. You don't check headroom in your tools. The pool drops agents gracefully when the context fills ‚Äî agents that reported results keep them, agents that didn't get recovered via `policy.onRecovery()`.
</Note>

## `useAgentPool` vs `runAgents`

Both create agent pools. The difference is branch lifetime.

<Tabs>
  <Tab title="runAgents">
    Branches are pruned when the pool completes. Use this when you only need findings.

    ```typescript
    const pool = yield* runAgents({ tasks, tools: toolMap });
    // Branches already pruned ‚Äî pool.agents[*].result still available
    ```
  </Tab>

  <Tab title="useAgentPool">
    Branches stay alive after the pool completes. Use this when you need to fork from agent branches ‚Äî for verification, follow-up generation, or additional extraction.

    ```typescript
    const pool = yield* useAgentPool({ tasks, tools: toolMap });
    // Branches still alive ‚Äî you can fork from pool.agents[0].branch
    const verified = yield* generate({ prompt, parent: pool.agents[0].branch });
    // Branches pruned when the enclosing scope exits
    ```
  </Tab>
</Tabs>

## Nested pools

A tool can spawn its own agent pool. The inner pool shares the same GPU and KV cache.

```
Outer pool:
  withSharedRoot (root, pos 0..866)
    ‚îú‚îÄ‚îÄ Agent A0 (fork at 866)
    ‚îÇ     ‚îî‚îÄ‚îÄ tool dispatch: web_research
    ‚îÇ           ‚îî‚îÄ‚îÄ withSharedRoot (inner root, fork from A0's branch)
    ‚îÇ                 ‚îú‚îÄ‚îÄ Sub-agent 0 (fork from inner root)
    ‚îÇ                 ‚îú‚îÄ‚îÄ Sub-agent 1 (fork from inner root)
    ‚îÇ                 ‚îî‚îÄ‚îÄ Sub-agent 2 (fork from inner root)
    ‚îú‚îÄ‚îÄ Agent A1 (fork at 866)
    ‚îî‚îÄ‚îÄ Agent A2 (fork at 866)
```

Inner and outer pools share the same `BranchStore`. Context pressure is global ‚Äî `cells_used` accounts for all branches across all nesting levels. When the inner pool exits, its branches are pruned and KV cells are freed for the outer pool. Inner sub-agents inherit the calling agent's full attention state via the warm path.

## Events

The pool emits events through the `Events` channel for every lifecycle transition:

- `agent:spawn` ‚Äî agent created and prefilled
- `agent:produce` ‚Äî token generated (with optional entropy/surprisal)
- `agent:tool_call` ‚Äî tool invoked
- `agent:tool_result` ‚Äî tool result received (with `contextAvailablePercent`)
- `agent:tool_progress` ‚Äî progressive tool feedback (e.g. reranker scoring)
- `agent:report` ‚Äî agent submitted result via terminal tool
- `agent:done` ‚Äî agent completed
- `agent:tick` ‚Äî batch decode committed (with KV pressure snapshot)

Subscribe in a spawned consumer to build TUI, logging, or monitoring:

```typescript
yield* spawn(function*() {
  for (const ev of yield* each(events)) {
    if (ev.type === "agent:report") {
      console.log(`Agent ${ev.agentId}: ${ev.result}`);
    }
    yield* each.next();
  }
});
```
