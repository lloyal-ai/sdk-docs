---
title: "Corpus Source"
description: "Local file research with grep, semantic search, read_file, and recursive agents"
---

`CorpusSource` turns a directory of files into a searchable knowledge base. Agents investigate it with four tools: semantic search, exhaustive grep, targeted file reads, and recursive sub-agent research.

## Setup

<Steps>
  <Step title="Load resources and chunk them">
    ```typescript
    import {
      CorpusSource,
      loadResources,
      chunkResources,
      createReranker,
    } from "@lloyal-labs/rig/node";

    const resources = loadResources("./docs");
    const chunks = chunkResources(resources);
    ```

    `loadResources` reads all files from the directory. `chunkResources` splits them into paragraph-level chunks with headings and line ranges.
  </Step>
  <Step title="Create the source">
    ```typescript
    const source = new CorpusSource(resources, chunks, {
      grep: {
        maxResults: 50,       // max matches returned (default 50)
        lineMaxChars: 200,    // max chars per matched line (default 200)
      },
      readFile: {
        defaultMaxLines: 100, // default range when no endLine specified (default 100)
      },
    });
    ```

    All options are optional with sensible defaults. The simplest form is `new CorpusSource(resources, chunks)`. At construction, `CorpusSource` wires up `ReadFileTool` and `GrepTool` immediately. These tools work without a reranker.
  </Step>
  <Step title="Bind the reranker">
    ```typescript
    yield* source.bind({ reranker });
    ```

    `bind()` tokenizes all chunks through the reranker and prepends a `SearchTool` to the tool list. Call it once before using the source.
  </Step>
</Steps>

<Note>
`bind()` is idempotent. Calling it multiple times is safe -- it skips work after the first call.
</Note>

## The 4 Corpus Tools

| Tool | What it does | When agents use it |
|---|---|---|
| `search` | Semantic search via reranker scoring. Returns ranked chunks with file names, headings, scores, and line ranges. | First tool agents reach for. Broad semantic matching. |
| `grep` | Exhaustive regex scan across all files. Returns matching lines with file names and line numbers, capped at 50. | When the agent needs exact pattern matches -- specific identifiers, error codes, function names. |
| `read_file` | Read file content at specific line ranges. Tracks per-agent read state to skip already-read lines. | After search or grep reveals a relevant section. Agents read the surrounding context. |
| (recursive delegate) | When `agentPool()` is called with `recursive: true`, a delegate tool is added to the toolkit. Sub-agents can spawn their own sub-agents for deeper investigation. | Configured by the harness, not the source. |

<Tip>
`search` is ordered first in the toolkit deliberately. The model prefers tools listed earlier, so agents default to semantic search before falling back to grep or read_file.
</Tip>

### Read tracking

`ReadFileTool` tracks which lines each agent has already read, keyed by `agentId:filename`. When an agent requests a range it has already seen, the tool returns only the unread portions. This prevents redundant context inflation.

### Recursion via `agentPool()`

When the harness calls `agentPool()` with `recursive: true`, a delegate tool is added to the agents' toolkit. Sub-agents can call it to spawn their own sub-agents for deeper investigation. The recursion shape (tool name, args schema, task extraction) is configured by the harness, not by the source.

Hard-cut agents (those that exhaust their turn limit without calling the terminal tool) get a forced extraction pass via the policy's `onRecovery()`. The pool forks from the agent's branch and runs a grammar-constrained generation to recover partial findings. Configure via `DefaultAgentPolicyOpts.recovery`.

## Quick Start

```typescript
import { call } from "effection";
import {
  reportTool,
} from "@lloyal-labs/rig";
import {
  CorpusSource,
  loadResources,
  chunkResources,
  createReranker,
} from "@lloyal-labs/rig/node";
import { initAgents, agentPool, DefaultAgentPolicy } from "@lloyal-labs/lloyal-agents";

// Load corpus
const resources = loadResources("./knowledge-base");
const chunks = chunkResources(resources);
const reranker = await createReranker("./models/reranker.gguf");
const source = new CorpusSource(resources, chunks);

// Inside an Effection scope:
const { session, events } = yield* initAgents(ctx);

yield* source.bind({ reranker });

// Create an agent pool with the source's tools
const pool = yield* agentPool({
  tasks: [{ content: "How does the allocation pool handle cleanup?" }],
  tools: [...source.tools, reportTool],
  systemPrompt: CORPUS_RESEARCH_PROMPT,
  terminalTool: "report",
  recursive: { name: "research", extractTasks: (a) => a.questions as string[] },
  policy: new DefaultAgentPolicy({ recovery: { prompt: REPORT } }),
});
```

## createTools Helper

For simpler cases where you do not need the full `CorpusSource` lifecycle, use the `createTools` function to build a toolkit directly:

```typescript
import { createTools } from "@lloyal-labs/rig";
import { createReranker } from "@lloyal-labs/rig/node";

const toolkit = createTools({
  resources,
  chunks,
  reranker,
});
// toolkit.toolMap has search, read_file, grep, report
// toolkit.toolsJson has the JSON schemas for the model
```

This builds a flat toolkit with `SearchTool`, `ReadFileTool`, `GrepTool`, and `ReportTool`. Use this when you want corpus tools in a custom toolkit without the `Source` abstraction.
