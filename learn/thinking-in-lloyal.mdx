You already know how to build agents — tools, prompts, multi-step pipelines. The new concept is **structured concurrency**: every concurrent operation is bound to the scope that created it and cannot outlive it.

This matters because you have GPU state. Branches, KV cells, sampler state. When something goes wrong, cleanup must happen in the right order, every time.

## Why not async/await?

Cloud agent frameworks use async/await because each step is a stateless HTTP request. If your process crashes, no GPU state is lost — there is no GPU state.

Local inference is different. You have a KV cache with token-level state per branch. Multiple agents share that cache through forked sequences. When a tool throws, an agent hits the context limit, or the user cancels — GPU resources must be freed in the correct order.

<Note>
`Promise.race()` leaves the loser running in the background. A detached Promise from a tool call can outlive the agent that spawned it. With cloud APIs, these are wasted HTTP requests. With local GPU state, they're memory leaks.
</Note>

## Generators instead of async/await

lloyal uses `function*` with `yield*` instead of `async` with `await`. The framework can interrupt a generator between yields, enabling cancellation and cleanup. An async function awaiting a Promise can't be interrupted until it settles.

The syntax maps directly:

```typescript
// Cloud API pattern
async function research(query: string) {
  const plan = await llm.chat({ messages, tools });
  const results = await Promise.all(
    plan.questions.map(q => llm.chat({ messages: [q], tools }))
  );
  return synthesize(results);
}

// lloyal pattern
function* research(query: string): Operation<string> {
  const result = yield* agent({ systemPrompt, task: query, schema: planSchema });
  const plan = JSON.parse(result.rawOutput);

  const pool = yield* agentPool({
    tasks: plan.questions.map(q => ({ content: q })),
    tools: [searchTool, reportTool],
    systemPrompt: RESEARCH_PROMPT,
    terminalTool: "report",
  });
  return pool.agents.map(a => a.result);
}
```

Key differences:

- `yield*` instead of `await` — runs an Operation within the current scope
- `Operation<T>` instead of `Promise<T>` — describes work without executing immediately
- `agentPool` instead of `Promise.all` — concurrent agents with shared GPU and automatic cleanup

<Tip>
The generator syntax looks unfamiliar for about an hour. After that, it reads like sequential code with automatic cleanup — which is exactly what it is.
</Tip>

## Scopes as lifetimes

Everything runs inside a scope. Scopes nest like function calls:

```
main()
 └─ initAgents()                         ← sets up Ctx, Store, Events
     └─ agentPool()                ← creates root, toolkit, agents
         └─ useAgentPool()               ← tick loop manages N agents
             ├─ agent 0 (generating)
             ├─ agent 1 (awaiting tool)
             │   └─ scoped(tool.execute)  ← tool runs in child scope
             │       └─ agentPool() ← recursive tool spawns sub-pool
             │           └─ useAgentPool()
             └─ agent 2 (generating)
```

When a scope exits — normally, via error, or via cancellation — every child scope halts and cleanup runs in reverse order. Inner branches are pruned. Inner roots are pruned. Tool error boundaries catch without crashing the outer pool. This happens automatically, every time.

## The five primitives

lloyal uses five Effection primitives. That's the entire concurrency surface area.

<Steps>
  <Step title="call() — bridge to Promises">
    Wraps a Promise-returning function for use inside a generator. This is how lloyal interacts with async APIs.

    ```typescript
    const tokens = yield* call(() => ctx.tokenize(prompt));
    yield* call(() => branch.prefill(tokens));
    ```

    `call()` handles Promises, Operations, and plain values uniformly.
  </Step>

  <Step title="ensure() — guaranteed cleanup">
    Registers a callback that runs when the scope exits, no matter how it exits.

    ```typescript
    const branch = parent.forkSync();
    yield* ensure(() => {
      if (!branch.disposed) branch.pruneSync();
    });
    // Branch is ALWAYS pruned — normal return, error, or cancellation
    ```

    This is how every agent branch is cleaned up. You never manually prune.
  </Step>

  <Step title="spawn() — concurrent work within a scope">
    Launches a concurrent operation bound to the current scope.

    ```typescript
    yield* spawn(function*() {
      for (const ev of yield* each(events)) {
        renderToUI(ev);
        yield* each.next();
      }
    });
    // Halted automatically when the parent scope exits
    ```
  </Step>

  <Step title="scoped() — error boundaries">
    Creates a child scope that catches errors without crashing the parent.

    ```typescript
    try {
      const result = yield* scoped(function*() {
        return yield* call(() => tool.execute(args));
      });
    } catch (err) {
      result.result = `Tool error: ${err.message}`;
      // Other agents survive
    }
    ```

    This is how the agent pool dispatches tools — each runs in its own `scoped()` boundary.
  </Step>

  <Step title="createContext() — scoped dependency injection">
    Creates a typed value available to all child operations without argument passing.

    ```typescript
    // Define
    export const Ctx = createContext<SessionContext>("lloyal.ctx");

    // Set (parent scope)
    yield* Ctx.set(sessionContext);

    // Read (any child scope)
    const ctx = yield* Ctx.expect();
    ```

    A tool doesn't need to know how many layers of pools it's nested inside. It reads from context and gets the right value.
  </Step>
</Steps>

## Mapping to familiar patterns

If you know any of these, lloyal's model is the same idea applied to GPU-backed inference:

| Concept | Swift | Kotlin | lloyal |
|---------|-------|--------|--------|
| Scoped concurrency | `withTaskGroup` | `coroutineScope` | `agentPool` |
| Spawn child task | `group.addTask` | `launch` | `spawn()` |
| Structured cleanup | Task cancellation propagates | `Job` hierarchy + `finally` | Scope exit halts children, `ensure()` runs |
| Scoped values | `TaskLocal` | `CoroutineContext` | `createContext()` |
| Error boundary | `withThrowingTaskGroup` | `supervisorScope` | `scoped()` |
| Async bridge | `withCheckedContinuation` | `suspendCoroutine` | `call()` |

<Tip>
If you've never used any of these, the core insight is: **a scope owns its children, and cleanup is automatic**. You don't manage branch lifecycles, agent cleanup, or KV pressure recovery manually.
</Tip>

## What this means in practice

### You never manually prune branches

```typescript
// Wrong — manual lifecycle management
const branch = parent.forkSync();
try { /* ... */ }
finally { branch.pruneSync(); } // What if you forget?

// How lloyal does it — scope-managed
const branch = parent.forkSync();
yield* ensure(() => {
  if (!branch.disposed) branch.pruneSync();
});
// Registered once. It WILL run. You can't forget.
```

### Tool errors don't crash the pipeline

A tool that throws inside `scoped()` is caught at the dispatch level. The agent is marked done. Other agents continue. No try/catch boilerplate in tool implementations.

### Cancellation propagates correctly

When the user hits Ctrl-C during a nested research run, halt propagates from `main()` through every scope. Inner branches pruned, inner roots pruned, outer agents halted, model disposed. Correct order, every time, no orphaned GPU state.

### Context replaces argument drilling

```typescript
// Set once in initAgents:
yield* Ctx.set(ctx);
yield* Trace.set(traceWriter);

// Read anywhere in the tree:
const ctx = yield* Ctx.expect();
const tw = yield* Trace.expect();
```

## Getting started

<Steps>
  <Step title="Read an example">
    Start with `examples/react-agent/` — the simplest entry point. Model loading, `initAgents`, event subscription, tool use.
  </Step>

  <Step title="Read a pipeline">
    Then `examples/deep-research-web/harness.ts` — plan, research, synthesize, evaluate. Each stage composes `agent` and `agentPool`.
  </Step>

  <Step title="Run with --trace">
    Produces a JSONL file showing every prompt, tool call, agent turn, and branch lifecycle. The best way to understand runtime behavior.
  </Step>

  <Step title="Modify a task prompt">
    Change a prompt file in `tasks/` and re-run. Pipeline structure stays the same; model behavior changes. This is where most iteration happens.
  </Step>

  <Step title="Add a tool">
    Extend `Tool`, implement `execute()`, add to the tools array. Your tool automatically gets scope management, error boundaries, and trace integration.
  </Step>
</Steps>
