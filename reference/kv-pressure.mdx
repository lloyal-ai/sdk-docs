Agent pools share a single KV cache. As agents generate tokens and prefill tool results, the cache fills. `ContextPressure` tracks how much room remains and enforces thresholds at multiple points in the tick loop to prevent out-of-memory crashes. When an agent is killed by pressure, an inline recovery protocol extracts its findings before the pool continues, using the `hardLimit` reserve as a parallel report budget.

## The budget model

`ContextPressure` reads from `SessionContext._storeKvPressure()`, which returns `{ nCtx, cellsUsed, remaining }` where `remaining = nCtx - cellsUsed`. Two thresholds partition `remaining` into three zones:

```
                nCtx
 ┌──────────┬───────────────────┬──────────┬────────┐
 │cellsUsed │    headroom > 0   │ softLimit│hardLim │
 │ (in use) │   (new work OK)   │(reserved)│(recov) │
 └──────────┴───────────────────┴──────────┴────────┘
              <- remaining ->
                                |          |
 headroom = remaining - softLimit          |
 critical = remaining < hardLimit ---------┘
```

- **headroom > 0** — room for new work. Tool results can be prefilled, agents can continue generating.
- **headroom ≤ 0** — over budget. SETTLE defers tool results. PRODUCE nudges the model to wrap up.
- **critical** — `remaining` has dropped below `hardLimit`. Agents are killed before `produceSync()` to prevent `llama_decode` crashes. The `hardLimit` reserve is the recovery extraction budget.

## The `hardLimit >= nBatch` invariant

`hardLimit` isn't an arbitrary safety margin — it's the **parallel report budget** for the recovery path. When `pressure.critical` fires and the kill proceeds, `recoverInline` prefills its extraction prompt and runs a single-agent produce/commit loop *inside* the reserved `hardLimit` cells. If that reserve is smaller than the context's `nBatch`, the first recovery decode fails with `llama_decode: failed to find a memory slot for batch of size N` — native OOM.

The pool enforces this at `useAgentPool` entry. If `hardLimit < nBatch`, the pool throws:

```
useAgentPool: Invariant Violation — hardLimit (128) must be >= nBatch (512).
Recovery reserves hardLimit cells for its own decode; if smaller than nBatch,
the next batch allocation will OOM. Increase policy.budget.context.hardLimit
to at least 512.
```

This is a compile-time/startup check, not a runtime surprise. Misconfigured policies fail loudly before any agents spawn.

<Note>
Until `SessionContext.nBatch` is exposed natively, the pool validates against `ContextPressure.ASSUMED_N_BATCH = 512` (llama.cpp's default `n_batch`). Once the native exposure lands, the check reads `ctx.nBatch` directly so policies running against a custom batch size validate against the actual value.
</Note>

## `ContextPressure` API

The class is constructed from a `SessionContext` and thresholds from the policy. It captures an immutable snapshot of KV state at construction time — frozen at phase boundaries so that all decisions within a phase are evaluated against the same baseline. Without this, items processed earlier in a loop would see different pressure than items processed later, making reject/nudge/kill decisions order-dependent and nondeterministic.

The pool reads thresholds from `policy.pressureThresholds` (which comes from `DefaultAgentPolicyOpts.budget.context`):

```typescript
const policy = new DefaultAgentPolicy({
  budget: { context: { softLimit: 1024, hardLimit: 1024 } },
});

pressure.nCtx              // total KV cache capacity (0 when no limit)
pressure.cellsUsed         // KV cells currently in use (monotonic within a pool run)
pressure.remaining         // nCtx - cellsUsed (Infinity when nCtx <= 0)
pressure.headroom          // remaining - softLimit
pressure.critical          // remaining < hardLimit
pressure.canFit(n)         // n <= headroom
pressure.percentAvailable  // 0–100 integer: remaining / nCtx * 100
```

`percentAvailable` is the single source of truth for the KV availability percentage. It is used by `policy.shouldExplore()` to decide explore vs exploit mode, recorded on `ToolHistoryEntry.contextAfterPercent` after each tool result settles, and injected into tool results as `_contextAvailablePercent` so agents can make pressure-aware decisions.

**Defaults:** `softLimit = 1024`, `hardLimit = 512` (matches llama.cpp's default `n_batch`). Pools that use recovery with a larger expected report body should raise `hardLimit` accordingly — e.g. `hardLimit: 1024` to fit a ~400-token scratchpad extraction.

## Enforcement points

`ContextPressure` is instantiated at multiple points in the tick loop. Each snapshot is fresh — constructed from current `cellsUsed` at that moment.

### 1. Prefill gate (`ctx.spawn`)

When the orchestrator calls `ctx.spawn(...)`, the pool tokenizes the agent's suffix (system + user + tool schemas + generation prompt) and checks `canFit`:

```typescript
const pressure = new ContextPressure(ctx, pressureOpts);
if (!pressure.canFit(suffixTokens.length)) {
  agent.branch.pruneSync();
  // pool:agentDrop reason=pressure_init
  throw new Error(`useAgentPool: cannot fit agent suffix under current pressure`);
}
```

Agents dropped at the prefill gate never emit `agent:spawn` events — they silently disappear before generating a single token. In parallel orchestration, this can mean fewer agents actually run than were requested; in chain, it means a mid-chain task can't start (which surfaces as an orchestrator error). In nested pools, the outer agents' live branches consume KV that the inner pool's prefill gate measures against.

### 2. PRODUCE critical (the kill threshold)

At the start of each tick, a fresh `ContextPressure` snapshot is taken. The per-agent PRODUCE loop consults `policy.shouldExit(agent, pressure)`:

```typescript
for (const a of agents) {
  if (a.status !== 'active') continue;

  const policyExit = policy.shouldExit?.(a, pressure);
  if (policyExit ?? pressure.critical) {
    // pool:agentDrop reason=pressure_critical (if driven by pressure)
    yield* recoverInline(a, policy, ctx, store, tw, ...);
    a.transition('idle');
    continue;
  }
  // ...produceSync + accumulate + commit
}
```

`DefaultAgentPolicy.shouldExit` returns `true` when `pressure.critical` fires and at most one kill per tick — the `_killedThisTick` flag prevents mass kills. Other agents survive to the next tick, where freed cells (from the just-killed agent's pruned branch) give them recovered headroom.

**Terminal-tool protection is narrowed to the graceful zone.** When `agent.currentTool === terminalTool` (the partial parser detected the model emitting a terminal-tool call like `report`) AND `!pressure.critical`, `shouldExit` returns `false` — protecting the agent from arbitrary time-limit kills mid-generation. But once `pressure.critical` fires, protection yields: the agent must release the KV before the next decode OOMs. This narrowing is what prevents the runaway-prose-into-terminal-tool pattern from exhausting the context.

### 3. PRODUCE soft-cut + budget nudge

When an agent hits a stop token and the parsed output contains a non-terminal tool call, the policy checks whether the call should be allowed:

```typescript
const overBudget = (a.turns >= maxTurns || pressure.headroom < 0)
  && (!terminalTool || tc.name !== terminalTool);
```

If `headroom < 0` (over the soft limit) and the tool call is not the terminal tool, the policy nudges the agent with a message *carrying the remaining token budget*:

```
KV memory pressure — report your findings now within 1100 words.
Turn limit reached — report your findings now within 1100 words.
Time limit reached — report your findings now within 1100 words.
```

The token ceiling is `pressure.remaining - pressure.hardLimit`, converted to a conservative word count (`floor(tokens * 0.7 / 10) * 10`, minimum 10) before being embedded in the message. **Words, not tokens**, because tokenizers vary across models but words are universal in training data — and under-advertising keeps the model within native decode headroom even if its word-to-token ratio differs from the 0.7 assumption. If the agent overshoots, `pressure.critical` fires on the next tick and recovery takes over (consuming the `hardLimit` reserve). If the agent obeys, the voluntary report lands cleanly.

Terminal tools always pass through without nudging — an agent that wants to report should never be blocked.

<Tip>
Model behavior varies: some small models honor the token budget closely, others produce verbose prose before their tool call. Both paths are covered. Voluntary report within budget → clean finish. Overshoot → kill + recovery report in the reserved cells. Either way, findings land on the spine. The budget-aware nudge just makes the happy path more likely.
</Tip>

### 4. SETTLE deferral (tool result gating)

After DISPATCH executes tools, the next tick's SETTLE phase drains those results. Each result is pressure-gated — but SETTLE never kills agents directly. Items that don't fit are deferred until headroom recovers:

```typescript
const settlePressure = new ContextPressure(ctx, pressureOpts);
let headroom = settlePressure.headroom;

for (const item of items) {
  const a = agentById.get(item.agentId);
  if (!a || a.status === 'idle') continue;

  if (item.prefillTokens.length > headroom) {
    deferred.push(item);  // retry next tick
    continue;
  }

  prefillPairs.push([a.branch, item.prefillTokens]);
  headroom -= item.prefillTokens.length;
}

return deferred;
```

The `headroom` variable is tracked locally because `cellsUsed` doesn't update until `store.prefill()` at the end of the phase. The local accounting ensures all items are evaluated against the same baseline.

Tool results are the largest KV consumers. A `web_search` result can be 2,000+ tokens. A `fetch_page` result can be 6,000+. SETTLE deferral is the primary backpressure mechanism during active research.

**Staggered exit in parallel orchestration.** In `parallel` orchestration, deferred items stay in `pendingSettled` across ticks. Headroom recovers when a sibling agent reports and its branch is pruned (`pruneOnReport: true` — default). Next tick, the deferred tool result against the fresh (larger) headroom fits. This is the staggered-exit pattern: one agent's big result is absorbed by another's completion.

**Spine growth in chain orchestration.** In `chain`, each task's findings accumulate on the shared root via `ctx.extendRoot` (typically 500–1,500 tokens per task). The spine grows monotonically; later tasks have less starting headroom than earlier ones. Size `hardLimit` and `softLimit` to accommodate the final task's constraints, not the first's.

**DAG orchestration** is a hybrid — dependent tasks inherit extended spine from predecessors; independent siblings stagger like parallel. Pressure behavior follows whichever property dominates at each node.

### 5. SETTLE stall-break (last resort)

If `deferred` has items but no active siblings can free KV (everyone is `awaiting_tool` or idle), the tick loop invokes the **stall-breaker**. This is where `policy.onSettleReject` is consulted:

```typescript
if (deferred.length > 0 && !agents.some(a => a.status === 'active')) {
  for (const item of deferred) {
    const a = agentById.get(item.agentId);
    const action = policy.onSettleReject?.(a, item.prefillTokens.length, stallPressure, config);

    if (action?.type === 'nudge') {
      // Replace the oversized result with a compact nudge payload
      // "Tool result too large for remaining KV. Report your findings now within N words."
      resolved.push(/* nudge SettledTool */);
      continue;
    }

    // action?.type === 'idle' or hook absent → drop
    const reason = action ? 'pressure_settle_reject' : 'settle_stall_break';
    tw.write({ type: 'pool:agentDrop', reason, ... });
    yield* recoverInline(a, ...);
    a.transition('idle');
  }
}
```

Three distinct `pool:agentDrop` reasons come out of this path:

- `pressure_settle_reject` — policy was consulted and returned `{type: 'idle'}`, OR returned `{type: 'nudge'}` but the nudge payload itself doesn't fit in the reserve. Policy-driven decision.
- `settle_stall_break` — `onSettleReject` hook is absent (legacy fallback). No policy input was available.

The `DefaultAgentPolicy.onSettleReject` returns `{type: 'nudge'}` when a terminal tool is configured and the agent has made at least one tool call — same budget message as the PRODUCE-phase nudge. The nudge replaces the oversized result, the agent's next generation is reaction-to-nudge (typically: report or retry with narrower query).

### `_contextAvailablePercent` in tool results

After each tool executes, the pool creates a fresh `ContextPressure` snapshot and injects `percentAvailable` into the tool result object as `_contextAvailablePercent`:

```json
{"matches": [...], "totalMatches": 4, "_contextAvailablePercent": 32}
```

The same value is emitted on the `agent:tool_result` event for TUI display and recorded on `ToolHistoryEntry.contextAfterPercent` for policy decisions.

<Note>
Nudge payloads do NOT currently include `_contextAvailablePercent` — the budget is carried in the *message string* ("within N words") instead. Word-based wording was chosen over tokens because tokenizer behavior varies across models but words are stable across training distributions. If you write a custom policy that produces nudge payloads, consider adding the percent field to the error object for symmetry with regular tool results.
</Note>

## Recovery extraction (trailing stop)

When an agent is killed by pressure, the pool extracts its accumulated findings **inline** before continuing the tick loop. This is the trailing stop — one agent sacrificed per critical tick, its findings extracted and KV freed, so remaining agents can continue.

### How it works

When `shouldExit` (or stall-break, or post-loop cleanup) kills an agent:

1. **Policy check**: Call `policy.onRecovery(agent, pressure)`. If it returns `{type: 'skip'}`, prune the branch and continue.
2. **Budget computation**: Policy computes `budget = max(50, pressure.remaining - prefillOverhead - batchBuffer)` and renders the recovery prompt's eta templates with `{budget}` context. The model receives its actual token budget.
3. **Grammar**: Create an eager JSON grammar from `jsonSchemaToGrammar({ result: string })`. Constrains from token 0 — the model MUST produce `{"result": "..."}`.
4. **Prefill**: Prefill the rendered extraction prompt into the agent's own branch. No fork — the agent's full KV context (system prompt, tool results, reasoning) is physically present.
5. **Set grammar**: `branch.setGrammar(reportGrammar)`. Eager mode — active from the first generated token.
6. **Generate**: Single-agent produce/commit loop until stop token.
7. **Parse + diagnostics**: Parse the JSON output. Emit `pool:recoveryProduce` with token count + output length, then one of:
   - `pool:recoveryReport` — parse succeeded, findings captured via `reportResult(parsed.result, 'scratchpad')`.
   - `pool:recoveryFailed` — parse failed. Event carries `reason` (`parse_error`, `no_result_field`, `scope_error`) and `outputExcerpt` (first 200 chars) so ops can diagnose without re-running.
8. **Prune**: Free the agent's branch. The freed KV cells give remaining agents headroom.

Recovery runs inside `scoped()` with error containment — if `llama_decode` fails during the produce/commit loop, the scope tears down cleanly and `pool:recoveryFailed` fires with the captured failure reason.

### Budget-aware recovery prompt

The recovery prompt is an eta template. `DefaultAgentPolicy.onRecovery` renders it with the current `budget` before returning to the pool. A minimal recovery prompt:

```eta
You are a research reporter. The agent has been terminated due to budget
constraints and must deliver findings now.

Call the report tool with the most important findings from the research above.
Prioritize in this order:
1. Direct quotes and specific evidence that answer the task
2. Key conclusions with source URLs / line references
3. Supporting context (only if budget allows)

IMPORTANT: You have approximately <%= it.budget %> words for your report. Be concise —
if you exceed the budget your report will be cut off.
---
Report your findings.
```

The `<%= it.budget %>` tag is substituted at `onRecovery` call time with the numeric budget. Prompts that don't use the tag render identically (backward compatible).

### Why eager grammar, not lazy

During normal generation, agents use lazy grammar with tool-call triggers — the model generates freely until `<tool_call>` fires, then the grammar constrains the tool call structure. Recovery uses **eager** grammar (`setGrammar`, not `setGrammarLazy`). The JSON schema grammar constrains from token 0: the model must produce `{"result":` followed by findings text. No tool calls possible, no model choice. This is the system extracting findings, not asking the model to report.

### Recovery and hardLimit

The `hardLimit` reserve exists specifically for recovery. Agents are killed when `remaining < hardLimit`. Recovery needs to fit:

- Extraction prompt prefill: ~100–200 tokens (short system + user turn).
- JSON `{"result": "..."}` generation: 200–500 tokens for a typical scratchpad report.
- Native batch workspace: ~512 cells (the `nBatch` floor).

Default `hardLimit = 512` satisfies the invariant but is tight for verbose reports. Policies expecting longer extractions should bump to 1024+.

<Note>
The recovery pressure check uses `remaining < promptTokens` (raw capacity), not `canFit` or `critical`. The `hardLimit` reserve was set aside precisely for this moment — checking `critical` would always fail because agents were just killed at the critical threshold.
</Note>

### Trailing stop behavior

The policy's `shouldExit` kills at most one agent per tick when pressure is critical. The `_killedThisTick` flag prevents mass kills. After recovery extracts and prunes, the freed KV gives remaining agents headroom to continue. If pressure stays critical, another agent is killed next tick.

### One-shot guard (post-loop)

If all agents are idle without results after the tick loop ends (e.g., killed by `maxTurns` or `free_text_stop`), a separate one-shot recovery pass extracts from any remaining unrecovered agents. This fires at most once per pool run via the `recoveryAttempted` flag.

### Scope

Recovery is per-pool. In chain/parallel/DAG pipelines, each pool runs its own recovery independently. For chain orchestration, recovered findings land on `agent.result` and the chain orchestrator's `extendRoot` call carries them to the spine — subsequent tasks inherit prior recovered findings via KV, same as voluntary reports.

## `cellsUsed` tracks live pressure

<Note>
`cellsUsed` increments on every `decode_each` (COMMIT, +1 per branch) and `decode_scatter` (SETTLE prefill, +tokens per branch). It **decrements** when a branch is pruned — each branch subtracts its unique cells (`position - fork_head`) on release. It resets on bulk operations: `retainOnly` sets it to the winner's position, `drain` resets to zero.

`cellsUsed` can exceed `nCtx` — there is no pre-decode guard in the native layer. Protection is entirely in the JS layer via `ContextPressure.critical` + the `hardLimit >= nBatch` invariant, which together guarantee the next decode has a valid batch allocation.

On `llama_decode` failure, `cellsUsed` is NOT rolled back (it remains at the post-increment value), but the branch's sampler, grammar, and metrics state IS rolled back via the native snapshot/restore mechanism.
</Note>

`remaining` and `percentAvailable` reflect actual KV usage and **recover** as branches are freed. When an agent reports and its branch is pruned (`pruneOnReport: true`), the freed cells are visible to the next `ContextPressure` snapshot. Remaining agents see increased headroom.

This has a direct impact on explore/exploit mode: `percentAvailable` can rise above the `exploreThreshold` after branches are pruned, switching surviving agents back to explore mode. The flip is intentional — it produces higher quality trajectories by allowing discovery when headroom recovers.

## Sizing thresholds

For **single-level pools** with default `nBatch=512`:

| Policy | softLimit | hardLimit | Rationale |
|---|---|---|---|
| Baseline (default) | 1024 | 512 | Minimum invariant-satisfying config |
| Research with recovery | 1024–2048 | 1024 | Generous recovery budget for scratchpad extraction |
| Chain pipeline (4+ tasks) | 2048 | 1024 | Spine accumulates across tasks — size for the last task's constraints |
| Nested / recursive pools | 3072 | 1024 | Outer pool reserves enough for inner pool's full lifecycle |

```typescript
const policy = new DefaultAgentPolicy({
  budget: { context: { softLimit: 2048, hardLimit: 1024 } },
});

yield* useAgentPool({
  root,
  orchestrate: chain(tasks, (task, i) => ({ /* ... */ })),
  tools: toolMap,
  maxTurns: 8,
  policy,
});
```

<Tip>
For recursive pipelines, the outer pool's `softLimit` must reserve enough for the inner pool's KV consumption. An inner pool with 3 agents consuming ~500 tokens each of shared prefix + tool results needs ~1500 tokens of headroom. Set the outer pool's `softLimit` accordingly.
</Tip>

Monitor with `trace: true` — the `[SETTLE]` and `[PRODUCE]` pressure logs print `remaining`, `headroom`, `cellsUsed`, and `nCtx` at each tick. Recovery logs `pool:recoveryProduce` / `pool:recoveryReport` / `pool:recoveryFailed`:

```
[PRODUCE] SOFT_LIMIT remaining=2200 headroom=-104 cellsUsed=14184 nCtx=16384
[SETTLE] DEFER read_file:1059 > headroom=-104
[SETTLE] NUDGE read_file: 33tok (replacing 1059)
[RECOVERY] prefilling agent=3, 143 tokens, remaining=1024
pool:recoveryProduce agent=3 tokenCount=218 outputLength=1204
pool:recoveryReport  agent=3 resultLength=1180
```

## Pressure-driven agent lifecycle

<Steps>
  <Step title="Pool setup">
    `hardLimit >= nBatch` invariant validated. Misconfigured policies throw at `useAgentPool` entry.
  </Step>
  <Step title="ctx.spawn">
    Prefill gate rejects agents whose suffix doesn't fit under softLimit. Agent dropped silently with reason `pressure_init`.
  </Step>
  <Step title="Each tick: PRODUCE critical">
    `pressure.critical` fires → `shouldExit` kills one agent → `recoverInline` extracts in the `hardLimit` reserve. Terminal-tool protection yields in critical.
  </Step>
  <Step title="Each tick: PRODUCE soft-cut">
    Non-terminal tool call with `headroom < 0` → policy nudges with "within N words" budget. Terminal tools pass.
  </Step>
  <Step title="Each tick: SETTLE deferral">
    Oversized tool results defer to next tick. Siblings freeing KV (parallel) or spine extension (chain) restores headroom.
  </Step>
  <Step title="Stall-break">
    If deferred items have no active siblings to free KV, policy's `onSettleReject` is consulted. Nudge → replace result with compact payload. Idle → drop with `pressure_settle_reject`. No hook → `settle_stall_break`.
  </Step>
  <Step title="Recovery + diagnostics">
    Every drop path runs `recoverInline` (unless `onRecovery` returns skip). Diagnostic events: `pool:recoveryProduce`, then exactly one of `pool:recoveryReport` or `pool:recoveryFailed`.
  </Step>
</Steps>

## Kill / nudge decision matrix

The decision to kill or nudge an agent happens at multiple phases, each with its own inputs and side-effect chain. Reading the code site-by-site makes the behavior hard to reason about — the following matrix enumerates every decision point.

### Phases

| Phase | Where | When |
|---|---|---|
| SPAWN | `ctx.spawn` prefill gate | Before agent runs |
| PRODUCE top-of-tick | `policy.shouldExit` | Each tick, before `produceSync` |
| PRODUCE at `isStop` | `policy.onProduced` | Agent emits stop token |
| SETTLE normal | `settle()` fit check | Each tick when items present |
| SETTLE stall-break | `policy.onSettleReject` | Deferred items + no active siblings |
| Pool close | cleanup loop | After tick loop exits |

### Decisions

| Phase | Condition | Outcome | Recovery? |
|---|---|---|---|
| SPAWN | suffix > headroom | drop `pressure_init`, `ctx.spawn` throws | no |
| PRODUCE top | `pressure.critical` + no `shouldExit` hook | drop `pressure_critical` | yes |
| PRODUCE top | `pressure.critical` + `_killedThisTick` already set | skip (one kill per tick) | no |
| PRODUCE top | `pressure.critical` + `currentTool === terminalTool` | drop `pressure_critical` (protection yields under critical) | yes |
| PRODUCE top | `!pressure.critical` + `currentTool === terminalTool` | protected, no drop | n/a |
| PRODUCE top | `shouldExit` returns `true` (time limit, custom) | drop `policy_exit` | yes |
| PRODUCE `isStop` | tool call + `headroom < 0` + terminal tool configured + `toolCallCount > 0` | nudge (PRODUCE, `_nudgedThisTick`) | no |
| PRODUCE `isStop` | tool call + `headroom < 0` + no terminal tool | drop `pressure_softcut` via `handleIdleDrop` | no (post-loop only) |
| PRODUCE `isStop` | `turns >= maxTurns` + terminal tool configured | nudge "Turn limit reached" | no |
| PRODUCE `isStop` | `turns >= maxTurns` + no terminal tool | drop `maxTurns` via `handleIdleDrop` | no (post-loop only) |
| PRODUCE `isStop` | terminal tool call | report captured (no drop) | n/a |
| SETTLE normal | item fits | prefill, transition to `active` | n/a |
| SETTLE normal | item doesn't fit + any active sibling | defer to next tick | n/a |
| SETTLE normal | item doesn't fit + no active siblings | invoke stall-break (below) | — |
| SETTLE stall-break | `onSettleReject` returns `nudge` + nudge fits | replace item with nudge payload, agent continues | no |
| SETTLE stall-break | `onSettleReject` returns `nudge` + nudge doesn't fit | drop `pressure_settle_reject` | yes |
| SETTLE stall-break | `onSettleReject` returns `idle` | drop `pressure_settle_reject` | yes |
| SETTLE stall-break | `onSettleReject` hook absent | drop `settle_stall_break` | yes |
| Pool close | agent idle without result + `onRecovery` returns `extract` | `recoverInline` only (no drop event) | yes |
| Pool close | agent idle without result + `onRecovery` returns `skip` | branch prune only | no |

### Hidden state interactions

- **`_killedThisTick`** gates `shouldExit` — one kill per tick in critical. Does NOT apply to SETTLE stall-break, which can drop multiple agents on the same tick.
- **`_nudgedThisTick`** gates `_handleOverBudget` — one nudge per tick in PRODUCE. Does NOT apply to SETTLE stall-break, which can emit multiple nudges per tick (one per deferred item).
- **Terminal-tool protection (`currentTool === terminalTool`)** is only checked in `DefaultAgentPolicy.shouldExit`. SETTLE stall-break drops and pool-close cleanups do NOT check `currentTool`.
- **`handleIdleDrop` (PRODUCE `isStop` kill) does NOT call `recoverInline`**. Agents dropped via this path stay idle; their findings are only extracted by the pool-close cleanup loop.

## Drop reasons

The `pool:agentDrop` trace event records the specific enforcement point:

| Reason | Origin | Meaning |
|---|---|---|
| `pressure_init` | `ctx.spawn` prefill gate | Agent suffix doesn't fit under softLimit |
| `pressure_critical` | PRODUCE `shouldExit` | `remaining < hardLimit`; recovery runs in reserve |
| `pressure_softcut` | PRODUCE `_handleOverBudget` | Non-terminal over-budget + no terminal tool available |
| `pressure_settle_reject` | SETTLE stall-break (policy-driven) | Policy returned `idle` or nudge that can't fit |
| `settle_stall_break` | SETTLE stall-break (legacy fallback) | `onSettleReject` hook absent |
| `time_exceeded` | PRODUCE shouldExit (time hardLimit) | Wall-clock hardLimit crossed |
| `policy_exit` | PRODUCE shouldExit (custom policy) | Policy returned `true` not driven by pressure |
| `maxTurns` | PRODUCE (policy `_handleOverBudget`) | Turn count ≥ maxTurns, no terminal tool |
| `tool_error` | DISPATCH | Tool's `execute()` threw |

Every drop reason (except `pressure_init`, which is pre-spawn) is followed by a recovery attempt whose outcome is visible in `pool:recoveryReport` or `pool:recoveryFailed` — no silent data loss.
