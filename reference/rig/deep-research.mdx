The deep-research example is the reference RIG pipeline. It chains six operations into a complete research workflow: decompose the query, investigate with source-specific agents, bridge between sources, evaluate findings convergence, synthesize (with grounding tools when agents disagree), and verify convergence.

## Pipeline Overview

```
Plan → Research → [Bridge →] Findings Eval → Synthesize → Eval
```

Each stage runs inside the [tick loop](/reference/concurrency). All agent coordination uses the same primitives: `withSharedRoot`, `useAgentPool`, `generate`, `diverge`.

## Stage Walkthrough

<Steps>
  <Step title="Plan">
    Grammar-constrained decomposition of the user query into sub-questions.

    The `PlanTool` uses `generate()` with a JSON schema grammar to produce structured output:

    ```json
    {
      "questions": [
        { "text": "What are the mechanisms of X?", "intent": "research" },
        { "text": "Can you clarify what you mean by Y?", "intent": "clarify" }
      ]
    }
    ```

    The router inspects the result:
    - **All research** -- decompose into parallel sub-questions
    - **All clarify** -- return clarification questions to the user
    - **Mixed** -- drop clarify, dispatch research
    - **Empty array** -- passthrough, investigate the original query directly

    Passthrough queries get double `maxTurns` since they run as a single focused agent.
  </Step>
  <Step title="Research">
    The harness calls `spawnAgents()` for each source. Agents interleave retrieval and generation — searching, reading, forming hypotheses, searching again.

    All agents within a source run concurrently on shared GPU compute. Sources run sequentially, each getting the full KV budget.

    ```typescript
    for (const source of opts.sources)
      yield* source.bind({ reranker });

    for (const source of opts.sources) {
      const scorer = source.createScorer(query);
      const pool = yield* spawnAgents({
        tools: source.tools,
        systemPrompt: RESEARCH_PROMPT,
        tasks: activeQuestions,
        terminalTool: { name: 'report', tool: reportTool },
        recursive: { name: 'web_research', extractTasks: (a) => a.questions as string[] },
        extractionPrompt: REPORT,
        pruneOnReport: true,
        scorer,
        echoThreshold: 0.8,
      });
      // Bridge between sources (see next step)
    }
    ```

    **Entailment scoring** steers agents toward productive threads. The `scorer` (created per source, per query via `source.createScorer()`) uses the reranker as a real-time cross-encoder. Three checks at the delegation boundary:

    - **Web search results** — reranked by entailment score before the agent sees them, demoting lexically similar but semantically irrelevant results
    - **Delegation entailment** — proposed sub-questions scored against the original query. Questions that don't entail are filtered before spawning inner pools.
    - **Echo detection** — surviving questions scored against the agent's own task. If all are paraphrases (similarity > `echoThreshold`), the delegation is rejected — the agent is nudged to report instead of re-delegating.

    **Content boundaries** (fetch_page chunks, corpus search results) behave differently based on explore/exploit mode:

    - **Explore mode** (default, KV > 40% available): scored against the **agent's local query only** — preserving serendipitous discovery of bridging content that connects the investigation to the answer.
    - **Exploit mode** (KV < 40% available): `scoreRelevanceBatch` applies dual scoring — `min(toolQueryScore, originalQueryScore)` per chunk. Tightens focus when headroom is low, at the cost of discovery. Agents report faster with focused content instead of consuming remaining KV on speculative reads.

    The threshold is controlled by `DefaultAgentPolicy.exploreThreshold` (default 40, representing `ContextPressure.percentAvailable`). The policy evaluates `shouldExplore()` at each DISPATCH with a fresh pressure snapshot — so an agent that starts exploring can transition to exploit mid-run as KV fills. The harness can also force exploit mode via `policy.setExploitMode(true)`.

    Exploit-mode activations are traced as `entailment:content:exploit` events with the pressure snapshot and both score flavors (`toolQueryScore` from the agent-local pass, `combinedScore` from `scoreRelevanceBatch`).

    Agents killed by context pressure are recovered via the pool's built-in scratchpad extraction (`extractionPrompt`). A confabulation guard skips agents with insufficient context (fewer than 100 tokens or fewer than 2 tool calls).

    Recursive delegation returns both primary results and `nestedResults` — inner sub-agent findings that would otherwise be lost when the outer agent cherry-picks during its own report.
  </Step>
  <Step title="Bridge">
    Runs between sources when multiple sources are configured and the completed source produced findings.

    A single agent structures discoveries into three tiers: what was established, where evidence is incomplete, what was not covered. These discoveries condition the next source's sub-questions. See [Bridges](/reference/rig/bridges) for details.
  </Step>
  <Step title="Findings Eval">
    Grammar-constrained evaluation of whether research agents agree on key claims. Returns `{converged, conflicts[]}`.

    A contradiction exists when one agent claims something IS the case with specific evidence while another claims it is NOT the case. Different coverage angles are complementary, not contradictory. An agent with direct quotes makes a stronger claim than one reporting "not found."

    ```typescript
    const findingsEval = yield* evaluateFindings(res.agentFindings, opts);
    // { converged: false, conflicts: ["Agent 1 claims X with quotes. Agent 2 found no X."] }
    ```

    This drives the next step — when findings converge, synthesis is straightforward. When they diverge, the synthesizer gets tools to verify.
  </Step>
  <Step title="Synthesize">
    A synthesis agent integrates findings from all sources into a structured report with source attribution.

    The synthesizer's prompt and tools adapt to inference state via an [Eta template](/reference/prompt-composition):

    - **Converged findings** — report-only toolkit, standard synthesis prompt. No grounding needed.
    - **Diverged findings** — grounding tools (search, read_file, grep) added to toolkit, conflicts section in prompt. The synthesizer reads the source directly to resolve each conflict. It determines which agent's interpretation is supported by the text — no majority voting.
    - **Multi-source** — source passages section included with citation instructions.
    - **Single-source** — source passages section omitted entirely, not filled with "(none)".
  </Step>
  <Step title="Eval">
    Multi-branch semantic comparison via `diverge()`.

    Fork N branches from a shared frontier. Generate independently with the same verify prompt. Check convergence.

    ```typescript
    const samples = yield* diverge({
      prompt,
      attempts: opts.verifyCount,
      params: { temperature: 0.7 },
    });
    ```

    A grammar-constrained evaluator compares the responses:
    - **Converged** -- branches agree, the model is confident
    - **Diverged** -- branches disagree, the answer needs refinement
  </Step>
</Steps>

## Warm vs Cold

The pipeline handles both cold starts (first query) and warm continuations (follow-up questions) automatically.

<Tabs>
  <Tab title="Cold (first query)">
    Research runs inside `withSharedRoot`. A shared prefix branch is created, agents fork from it. After synthesis, `promoteTrunk` creates a new trunk with the query-response pair for future warm turns.
  </Tab>
  <Tab title="Warm (follow-up)">
    Prior conversation tokens are physically present in the trunk's KV cache. Research runs against the existing trunk via `warmResearch` -- sources bind and execute without creating a new shared root. After synthesis, `appendTurn` prefills the new query-response pair onto the existing trunk.
  </Tab>
</Tabs>

## CLI Usage

Three configurations depending on your data sources:

<Tabs>
  <Tab title="Web only">
    ```bash
    TAVILY_API_KEY=tvly-... npx tsx examples/deep-research-web/main.ts \
      ./models/your-model.gguf \
      --reranker ./models/reranker.gguf \
      --query "What are the latest advances in protein folding?"
    ```
  </Tab>
  <Tab title="Corpus only">
    ```bash
    npx tsx examples/deep-research-web/main.ts \
      ./models/your-model.gguf \
      --corpus ./docs \
      --reranker ./models/reranker.gguf \
      --query "How does the allocation pool handle cleanup?"
    ```
  </Tab>
  <Tab title="Web + Corpus">
    ```bash
    TAVILY_API_KEY=tvly-... npx tsx examples/deep-research-web/main.ts \
      ./models/your-model.gguf \
      --corpus ./docs \
      --reranker ./models/reranker.gguf \
      --query "Compare our implementation with the latest research"
    ```

    Sources run sequentially: corpus first, then web. A [bridge](/reference/rig/bridges) structures corpus discoveries before web research begins.
  </Tab>
</Tabs>

### Options

| Flag | Description | Default |
|---|---|---|
| `--query <text>` | Run a single query and exit | Interactive REPL |
| `--corpus <dir>` | Load local files as a corpus source | None |
| `--reranker <path>` | Path to reranker model (GGUF) | `models/qwen3-reranker-0.6b-q4_k_m.gguf` |
| `--findings-budget <n>` | Max chars for reranked passages | 4000 |
| `--trace` | Write structured trace events to `trace-*.jsonl` | Off |
| `--verbose` | Show stderr debug output | Off |
| `--jsonl` | Machine-readable JSON Lines output | Off |

### Defaults

The pipeline runs with 3 research agents, 3 verify branches, and 20 max tool turns per agent. Context size defaults to 16384 tokens (configurable via `LLAMA_CTX_SIZE`).

<Tip>
Drop `--query` to enter the interactive REPL. Ask follow-up questions -- the pipeline automatically switches to warm mode and continues the conversation with full prior context in KV.
</Tip>
